## 1. 引言：当容器只剩一扇窗

两个月前，我接手了一个让人抓狂的任务：在一个只允许 HTTP/HTTPS 出站的容器里，实现远程交互式 Shell。

这听起来像是玩笑，但真实场景往往比玩笑更荒诞。这个容器是某 AI 助手的执行沙箱——为了安全，它切断了所有入站端口，封锁了原始 TCP 出站，甚至连 ping 都被无情地挡在门外。你能用的，只有 bash、python3、curl，以及一个名为 `http_proxy=http://127.0.0.1:18080` 的环境变量。

我尝试了所有常规方案：

- **cloudflared** 快速隧道 → Cloudflare API 域名被阻断
- **bore** 公共中继 → GitHub CDN 都连不上
- **serveo** SSH 反向隧道 → TCP 出站被全封
- **反向 SSH** 到自己的 VPS → VPS 都 ping 不通

每一扇门都焊死了，每一扇窗都贴着"此路不通"。

直到我注意到那个被忽略的细节：`echo $http_proxy`。原来，他们自己的 AI 也需要访问外网。为了让它能拉代码、调 API，平台不得不留了一个 HTTP 代理出口。这是他们给自己留的门，而我决定沿着这门走进去。

于是，便有了 wsstunnel —— 一个从绝境中生长出来的项目。如今它开源了，希望能帮到每一个曾被沙箱"关住"的你。

本文将从背景、技术实现、开发哲学、项目意义、推广策略五个维度，对 wsstunnel 进行一次全面的深度剖析。

## 2. 背景分析：受限网络环境的真实挑战

### 2.1 沙箱的"铁壁"

现代云计算和容器技术带来了极大的便利，但也引入了严格的安全限制。在线 IDE（如 GitHub Codespaces、Gitpod）、CI/CD Runner（如 GitHub Actions、GitLab CI）、AI 执行沙箱等环境，通常会实施以下限制：

- **无入站端口**：容器没有公网 IP，或者所有入站端口被防火墙阻断
- **出站白名单**：只允许 HTTP/HTTPS（80/443）出站，有时甚至只允许特定域名
- **强制 HTTP 代理**：所有出站流量必须经过一个公司或平台统一的 HTTP 代理服务器
- **无 root 权限**：无法安装系统级软件、无法修改网络配置
- **极简基础镜像**：可能只有 bash、python3、curl，连 ssh、nc 都没有

这种环境的初衷是安全的：防止恶意代码外传数据、防止攻击者反向控制容器。但对开发者来说，这却成了一座无法逾越的孤岛。

### 2.2 现有工具的集体失灵

在 wsstunnel 之前，市面上已有大量内网穿透和远程 Shell 工具，但它们在上述极端场景下纷纷失效：

| 工具 | 失败原因 |
|------|---------|
| ssh -R | 需要 TCP 直连出站，HTTP 代理无法穿透 |
| frp | 同样依赖 TCP 直连，且客户端需要配置文件 |
| ngrok | 依赖第三方服务，且免费版限制多；同样需要 TCP 出站 |
| cloudflared | 需要 Cloudflare API 连通，且依赖 TCP |
| chisel | 支持 HTTP 代理，但配置复杂，且没有 PTY 支持 |
| serveo / bore | 公共中继，域名可能被墙，且 TCP 出站受阻 |

这些工具的共性问题是：它们都假设你能发起原始的 TCP 连接。而当网络管理员强制所有流量走 HTTP 代理时，TCP 直连就死了。

### 2.3 唯一的那扇窗：HTTP 代理

HTTP 代理的工作原理是：客户端发送 `CONNECT host:port HTTP/1.1` 请求，代理服务器建立到目标服务器的 TCP 隧道，然后客户端在这个隧道里发送任意数据（如 TLS 握手、WebSocket 升级请求）。

这意味着：**只要你能通过 HTTP 代理发送 CONNECT 请求，你就能在代理背后建立任何 TCP 连接。**

这正是 wsstunnel 的突破口。它利用 `websocket-client` 库对 HTTP 代理的原生支持，将 WebSocket 握手封装在 CONNECT 隧道中，从而在纯 HTTP 出站环境中打通了一条双向实时通道。

## 3. 技术架构与核心实现

### 3.1 三方中继模型

wsstunnel 采用经典的三方中继架构：

```
前端（Frontend）         中继（Relay）          后端（Backend）
  操作者                   VPS 公网服务         目标受限设备
    │                         │                       │
    │── WebSocket ──────────►│◄── WebSocket ─────────│
    │   (AUTH:token)         │   (IAM_BACKEND:token) │
    │                         │                       │
    │── "whoami" ──────────►│── "whoami" ──────────►│
    │                         │                       │  bash 执行
    │◄── output ────────────│◄── output ────────────│
```

- **中继（Relay）**：运行在公网 VPS 上，使用 `websockets` 异步库，负责认证、多后端管理、消息路由
- **后端（Client）**：运行在受限容器内，使用 `websocket-client` 同步库，通过 HTTP 代理穿透，启动 PTY shell 并转发 I/O
- **前端（Frontend）**：任何 WebSocket 客户端（浏览器、websocat、Python 脚本），连接到中继后发送命令

这种架构的精髓在于：**所有连接都由客户端主动发起**，完美绕过了 NAT 和防火墙的入站限制。

### 3.2 协议走私：HTTP CONNECT 降维打击

这是 wsstunnel 穿透能力的核心。在客户端，我们使用 `websocket-client` 的 `http_proxy_host` 和 `http_proxy_port` 参数：

```python
ws.connect(
    server_url,
    http_proxy_host=proxy_host,
    http_proxy_port=proxy_port,
)
```

底层发生的事情：

1. 客户端向代理发送 `CONNECT your-vps:443 HTTP/1.1`
2. 代理与 VPS 建立 TCP 连接，返回 `200 Connection established`
3. 客户端在这个 TCP 隧道内发送 WebSocket 握手请求（`GET / HTTP/1.1 Upgrade: websocket`）
4. 握手成功后，WebSocket 帧在隧道内传输

从防火墙角度看，这只是一次普通的 HTTPS 代理请求；从代理服务器角度看，它只是在做 TCP 转发。但实际上，你在里面跑了一个完整的交互式 Shell。

这种技术被称为**协议走私（Protocol Smuggling）**——用标准协议的合规行为，绕过安全策略的检查。

### 3.3 PTY 模式：交付真终端

普通的反向 Shell 使用 `subprocess.PIPE` 与 bash 通信，这会导致 bash 检测到自己没有连接真实终端，从而禁用行编辑、禁用 TUI 程序（`vim`、`top` 直接崩溃）。

wsstunnel 默认使用 PTY（伪终端）模式：

```python
master_fd, slave_fd = pty.openpty()
_set_winsize(master_fd, rows, cols)
shell_proc = subprocess.Popen(
    [shell, "-i"],
    stdin=slave_fd, stdout=slave_fd, stderr=slave_fd,
    preexec_fn=os.setsid,
)
```

- `pty.openpty()` 创建一对伪终端文件描述符，让 bash 以为自己连接着一个物理终端
- `preexec_fn=os.setsid` 让 bash 成为新会话的组长，方便后续信号控制
- `__RESIZE:rows,cols` 命令通过 `fcntl.ioctl(master_fd, termios.TIOCSWINSZ)` 动态调整窗口大小

配合二进制帧透传，wsstunnel 完美支持 `vim`、`top`、`htop` 等 TUI 程序，体验接近原生 SSH。

### 3.4 多后端路由与集群管理

一个中继只连一个容器太浪费，wsstunnel 支持多个后端同时在线：

- 每个后端通过 `--name` 注册（自动分配或自定义名称）
- 前端发送 `LIST` 查看所有后端及模式（pty/pipe）、运行时间
- `USE <name>` 切换当前默认后端
- `@name <cmd>` 临时向指定后端发送命令
- 后端输出自动加上 `[@name]` 标签，多后端时区分来源

### 3.5 文件传输协议：base64 + 分块

v0.18.0 起，wsstunnel 支持在前后端之间直接传输文件。采用 base64 编码 + 64KB 分块的设计，路径通过 base64 编码避免特殊字符冲突：

```
上传流程：
  F→B: __FILE_BEGIN:{b64path}:{size}
  B→F: __FILE_OK:{b64path}:{size}        (确认)
  F→B: __FILE_CHUNK:{b64path}:{idx}:{b64}
  F→B: __FILE_END:{b64path}
  B→F: __FILE_DONE:{b64path}:{size}       (完成)

下载流程：
  F→B: __FILE_DOWNLOAD:{b64path}
  B→F: __FILE_BEGIN:{b64path}:{size}       (开始)
  B→F: __FILE_CHUNK:{b64path}:{idx}:{b64}
  B→F: __FILE_END:{b64path}:{size}         (完成)
```

### 3.6 四层保活体系

wsstunnel 构建了完整的连接保活体系：

| 层级 | 机制 | 间隔 | 位置 |
|------|------|------|------|
| WebSocket 协议层 | 禁用内置 ping（与自定义心跳互斥） | — | Relay |
| 应用层心跳 | `__PING__` / `__PONG__` | 30 秒 | Client → Relay |
| 断线重连 | 指数退避（5s → 300s 封顶） | 动态 | Client |
| Shell 重生 | PTY 崩溃后自动重启 | 最多 5 次 | Client |

### 3.7 嵌入式 Web 终端

Relay 内置 HTTP 服务器，直接请求 `http://vps:port` 即可获得基于 xterm.js 的全功能 Web 终端。页面在启动时从 `web/index.html` 加载到内存，后续请求直接返回缓存。支持多后端集群管理面板、文件上传、移动端浮窗控制键。

## 4. Hacker 技巧深度盘点

### 4.1 协议走私：把代理变成盲眼隧道

**问题**：目标容器仅允许 HTTP 出站，禁止任何裸 TCP 或 UDP 连接。

**技巧**：利用 `websocket-client` 对 HTTP 代理的底层支持，把本应用来缓存网页的代理服务器变成一个盲眼 TCP 隧道。

### 4.2 本地 MitM：按键嗅探与指令劫持

**问题**：用户想在 Shell 里下载文件，但 bash 不支持。

**技巧**：Client 在 PTY 接收循环里监听每一个按键，检测到 `dl <path>` 时直接劫持指令，代替 bash 执行，并在 bash 中注入 Ctrl+C 清空输入行：

```python
if ch in ("\r", "\n"):
    line = _key_buffer.strip()
    if line.startswith("dl "):
        os.write(master_fd, b"\x03\r")  # Ctrl+C
        threading.Thread(target=_send_file, ...).start()
```

### 4.3 状态影子追踪：克隆 Shell 的内心世界

**问题**：父进程无法读取子进程 bash 的当前工作目录（CWD）。

**技巧**：在输入流中拦截 `cd` 命令，在 Python 内存里维护一份 CWD 的影子副本：

```python
def _update_cwd(cmd):
    if cmd.startswith("cd "):
        target = ...
        _cwd = os.path.normpath(os.path.join(_cwd, target))
```

配合 Web 终端 JS 发送 `__CWD:<dir>` 控制消息，即使在后端按键缓冲模式下也能同步。

### 4.4 内核级欺骗：虚拟 TTY 的艺术

用 `pty.openpty()` 让 Linux 内核以为 bash 面前坐着一个拿着物理键盘的人。`preexec_fn=os.setsid` 让 bash 成为新会话组长——内核以为这个进程组连接着真正的终端。普通反向 Shell 跑 `vim` 直接崩溃，而这里 `vim` 能全屏运行。

### 4.5 协议信道复用：一根线上跑多个服务

利用 WebSocket 原生的帧类型机制，单连接上同时传输 PTY 画面、控制信号、文件数据：

```python
# 二进制帧 → PTY 画面
if isinstance(message, bytes):
    await _forward_binary_to_frontends(...)

# 文本帧 → 控制信令 + 文件数据
if isinstance(message, str) and message.startswith("__FILE_"):
    await _forward_to_frontends_untagged(...)
```

### 4.6 DOM 层事件劫持

绕过 xterm.js 的顶层 API，直接潜入 `beforeinput` DOM 事件处理移动端中文输入，配合 300ms 去重锁，从外部给黑盒打热补丁。

## 5. 工程化演进：从脚本到生产级系统

### 5.1 v0.9.0 重构：从闭包到类

早期版本使用 Python 闭包管理共享状态。v0.9.0 重构为 `RelayState` 类，状态变量显式声明在 `__init__` 中，类型注解明确，方法签名自文档化：

```python
class RelayState:
    backends: dict[str, Any]
    backend_modes: dict[str, str]
    frontends: set[Any]
    frontend_targets: dict[Any, str | None]
    frontend_text_modes: dict[Any, bool]
```

### 5.2 从零到 103 个测试

项目从零测试起步，目前已有 103 个测试覆盖协议解析、信号映射、PTY 窗口、广播函数、文件传输、CLI 参数等核心逻辑：

```bash
pytest  # 103 passed
```

### 5.3 性能优化

管道模式采用逐字节读取 + 智能行缓冲策略：

```python
buf = bytearray()
while True:
    byte = shell_proc.stdout.read(1)
    if byte == b"\n" or len(buf) >= 4096:
        ws.send(buf.decode("utf-8", errors="replace"))
        buf.clear()
```

平衡实时性和效率：逐字节确保不遗漏，行边界保证整洁，4096 上限防止长行延迟。

### 5.4 进程管理

优雅终止子进程：先 SIGTERM，超时后 SIGKILL：

```python
def _terminate_process(proc, timeout=5.0):
    proc.terminate()
    try:
        proc.wait(timeout=timeout)
    except subprocess.TimeoutExpired:
        proc.kill()
```

## 6. 开发哲学与设计原则

### 6.1 极简主义

- 不引入任何不必要的依赖
- 不实现"可能以后会用到"的功能
- 单文件瘦到极致，每个函数只做一件事

### 6.2 向后兼容

- Token 可选：无 token 时不校验
- 命名可选：自动分配 `backend-N`
- 标签可选：单后端时不加 `[@name]`
- 模式可选：旧客户端不发送 mode 时默认为 pipe

### 6.3 纯函数优先

协议解析函数设计为纯函数，不依赖任何外部状态：

```python
def _parse_backend_auth(msg, token) -> tuple[str, str] | None:
    """纯函数：输入消息+token，输出(name, mode)或None"""
```

### 6.4 渐进式重构

从单文件脚本到标准包，从闭包到类，从无测试到 103 个测试——每一步重构都有明确的业务驱动力，不为了"优雅"而重构。

### 6.5 防御性编程

- 每次广播前遍历连接集合并清理失效连接
- `finally` 块确保任何退出路径都清理共享状态
- 线程间通过 `threading.Event` 安全协调退出

### 7.1 与主流方案的对比

| 特性 | wsstunnel | ngrok | frp | chisel |
|------|-----------|-------|-----|--------|
| HTTP 代理穿透 | ✅ 原生支持 | ❌ | ❌ | ⚠️ 需手工配置 |

| 自托管 | ✅ | ❌ | ✅ | ✅ |

### 7.2 被低估的价值：轻量级边缘 C2/堡垒机

wsstunnel 的真正价值不止于"穿透"，而在于它提供了一个**可从零搭建的轻量级边缘管理框架**：

- 极低资源占用（< 30MB），适合嵌入式设备
- 完全自主可控，数据不经过第三方
- 所有流量明文可见，可审计
- 可扩展为通用 TCP 隧道

### 7.3 适用场景矩阵

| 场景 | 网络限制 | wsstunnel 的角色 |
|------|---------|-----------------|
| 在线 IDE 沙箱 | 仅 HTTP 出站 | 远程获取沙箱 Shell |
| CI/CD Runner | 受限容器网络 | 远程调试 CI 环境 |
| IoT 设备 | 低资源、无公网 IP | 轻量级远程管理 |
| 安全渗透测试 | DPI、协议白名单 | 流量伪装为 HTTP |

wsstunnel 从一次"被迫自救"的技术挑战中诞生。它不追求宏大叙事，只解决一个具体问题——在只能 HTTP 出站的容器里拿到一个真正的 Shell。

这个过程中沉淀下来的不仅是一个工具，更是一套应对极端网络限制的方法论：理解协议、寻找突破点、用最小的代价解决问题。如果你也曾被类似的限制困住，希望 wsstunnel 能帮上忙。而如果你有更好的点子，欢迎来 GitHub 提 issue 或 PR。
