# wsstunnel 深度解析：从零到一的 WebSocket 远程 Shell 中继架构

> 一款以 ~1900 行 Python 代码构建的生产级远程 Shell 隧道工具，具备 HTTP 代理穿透、PTY 交互式终端、多后端路由、文件传输、TLS 加密、心跳保活与指数退避重连能力。

---

## 一、项目概述与定位

### 1.1 项目定义

**wsstunnel**（PyPI 包名 `wsstunnel`）是一个基于 WebSocket 协议的远程 Shell 中继工具，专为穿透受限网络环境而设计。它的核心能力是：在目标设备仅能发起 HTTP 出站连接的极端网络条件下，仍能建立一条稳定的双向交互式 Shell 通道。

这个项目解决了一个非常具体的痛点——当你面对一个严格受限的容器环境（如在线 IDE 沙箱、CI/CD runner、受限办公网络中的设备），该环境仅允许通过 HTTP 代理访问外部网络，不允许任何直接的 TCP 连接，也不允许 UDP、ICMP 等协议出站。在这样严苛的条件下，传统的 SSH、frp、ngrok 等方案统统失效，而 wsstunnel 通过 HTTP CONNECT 代理隧道 + WebSocket 协议伪装的组合拳，优雅地突破了这一限制。

## 二、系统架构深度剖析

### 2.1 三方架构模型

wsstunnel 采用了经典的三方中继（Relay）架构，这也是 NAT 穿透和内网穿透领域最成熟的设计模式之一：

```
 Frontend（前端）          Relay（中继）        Backend（后端）
  第三方操作者               VPS 公网服务         目标受限设备
        │                      │                     │
        │── WebSocket ────────►│◄── WebSocket ───────│
        │                      │                     │
        │   发送命令            │   注册 + 执行        │
        │   接收输出            │   返回结果            │
        │                      │                     │
    直连 Internet             公网 VPS          仅 HTTP 代理出站
```

这种架构的设计精髓在于：**所有连接都由客户端主动发起**。Backend（位于受限网络中的目标设备）主动向 Relay（部署在公网 VPS 上的中继服务）建立 WebSocket 连接，绕过了 NAT 和防火墙的入站限制。Frontend（操作者）同样主动连接 Relay，通过 Relay 的路由机制与 Backend 间接通信。

### 2.2 为什么选择中继模式而非 P2P？

在远程控制领域，存在两种主要架构模式：

**P2P 直连模式**（如 chisel 的某些配置）：两端尝试直接建立连接，通常需要 STUN/TURN 服务器辅助 NAT 打洞。优点是延迟低，缺点是在对称 NAT 和严格防火墙环境下成功率低，且实现复杂度高。

**中继模式**（wsstunnel 的选择）：所有流量经过一个中间节点转发。虽然增加了一跳延迟，但连接成功率接近 100%，且实现简洁。对于远程 Shell 这种交互式场景，几十毫秒的额外延迟完全可以接受。

wsstunnel 选择中继模式是深思熟虑的结果：目标设备处于极端受限网络中，P2P 打洞几乎不可能成功（HTTP 代理只允许 HTTP CONNECT 方法，不支持 UDP 和裸 TCP），而中继模式下 Backend 只需发起一个看似普通 HTTP 请求的 WebSocket 连接，巧妙地利用了代理服务器的 CONNECT 隧道能力。

### 2.3 组件职责详解

#### 2.3.1 Relay（中继服务）—— `relay.py`（809 行）

Relay 是整个系统的中枢神经，承担以下核心职责：

**1) 连接管理与角色识别**

Relay 使用 `websockets` 异步库（基于 Python asyncio）监听 WebSocket 连接。每个新连接进入后，首先等待第一条消息（30 秒超时），根据消息内容判断连接身份：

```python
first = await asyncio.wait_for(websocket.recv(), timeout=30)
backend_info = _parse_backend_auth(first, token)
if backend_info:
    # 注册为后端
elif _is_frontend_auth(first, token):
    # 注册为前端
else:
    # 认证失败，断开
```

这种"首条消息决定角色"的设计极为精简——无需复杂的握手协议、无需额外的 API 端点、无需证书区分，一条消息搞定身份认证和角色注册。

**2) 后端注册与自动命名**

后端连接通过发送 `IAM_BACKEND:<token>:<name>:<mode>` 格式的消息完成注册：

- **自动命名**：`IAM_BACKEND:<token>` → 系统自动分配 `backend-N`
- **自定义命名**：`IAM_BACKEND:<token>:my-box:pty` → 指定名称 + 模式
- **模式标记**：`pty`（默认，支持 TUI 程序）或 `pipe`（向后兼容）

注册后广播后端列表给所有前端，每个前端都能看到实时连接状态。

**3) 前端消息路由**

前端消息处理是 Relay 的智能路由核心，支持四种路由模式：

| 路由模式 | 消息格式 | 行为 |
|---------|---------|------|
| 列表查询 | `LIST` | 返回所有后端名称、模式、运行时间 |
| 粘性切换 | `USE <name>` | 设置默认后端，后续命令直达 |
| 命名路由 | `@name <cmd>` | 临时发给指定后端，不影响当前选择 |
| 默认路由 | `<cmd>` | 发给当前目标后端 |

**4) 输出广播与标签**

多后端时输出自动加 `[@name]` 标签；单后端时保持原样（向后兼容）。文件传输消息（`__FILE_*`）走 `_forward_to_frontends_untagged` 路径，避免标签破坏协议格式。

**5) 双层心跳**

- **协议层**：Relay 显式禁用 `websockets` 的内置 ping（`ping_interval=None, ping_timeout=None`），避免与客户端的自定义心跳冲突
- **应用层**：后端发 `__PING__` → Relay 回 `__PONG__`，30 秒间隔

**6) TLS 加密**

支持三种部署模式：直接证书（Let's Encrypt）、自签名（`--insecure`）、nginx 反向代理（推荐生产）。

**7) 微信推送通知**

后端上线/下线时通过 `--wxpush url:key` 发送微信通知，使用 `httpx` 异步 HTTP 客户端。

**8) URL Token 快速认证**

前端连接时在 URL 中携带 `?token=xxx`，Relay 自动完成认证并返回 `AUTH_OK`，无需手动发送 AUTH 消息：

```python
url_token = self._extract_url_token(websocket)
if url_token and self.token and url_token == self.token:
    await websocket.send("AUTH_OK")
    await self._register_frontend(websocket)
    return
```

**9) 嵌入式 Web 终端**

Relay 内置 HTTP 处理器，在浏览器访问 `http://vps:port` 时返回 xterm.js Web 终端页面。页面在 Relay 启动时从 `web/index.html` 加载到内存，后续请求直接返回缓存。

#### 2.3.2 Client（后端客户端）—— `client.py`（655 行）

Client 是部署在目标设备上的轻量级代理，承担以下职责：

**1) HTTP 代理穿透**

使用 `websocket-client` 同步库（原生支持 HTTP CONNECT）：

```python
ws.connect(server_url,
    http_proxy_host=proxy_host,
    http_proxy_port=proxy_port,
)
```

从防火墙角度看，这只是一次普通的 HTTPS 代理请求。

**2) 指数退避重连**

```python
attempt = 0
while True:
    try:
        ws.connect(server_url)
        attempt = 0
    except Exception:
        attempt += 1
        delay = min(reconnect_interval * 2^(attempt-1), 300)
        time.sleep(delay)
```

从 5 秒开始，每次翻倍，300 秒封顶。

**3) PTY 模式（默认）**

使用伪终端，支持 vim/top/htop 等 TUI 程序：

```python
master_fd, slave_fd = pty.openpty()
shell_proc = subprocess.Popen(
    [shell, "-i"],
    stdin=slave_fd, stdout=slave_fd, stderr=slave_fd,
    preexec_fn=os.setsid,
)
```

关键能力：
- 窗口大小同步（`__RESIZE:<rows>,<cols>`）
- 信号转发（`__SIGNAL:SIGXXX`）
- Shell 崩溃自动重生（最多 5 次）
- 二进制帧透传，完整保留终端输出格式

**4) 管道模式（`--no-pty`）**

向后兼容，使用普通管道和行缓冲输出：

```python
shell_proc = subprocess.Popen(
    [shell, "-i"],
    stdin=subprocess.PIPE, stdout=subprocess.PIPE,
    stderr=subprocess.STDOUT, bufsize=0,
)
```

**5) 文件传输**

Client 内置文件上传和下载处理能力：

- **上传**：前端发送 `__FILE_BEGIN:{b64path}:{size}` → Client 创建文件 → 接收 `__FILE_CHUNK:` 数据块 → `__FILE_END:` 完成
- **下载**：前端发送 `__FILE_DOWNLOAD:{b64path}` 或在 Shell 中输入 `dl <path>` → Client 读取文件分块发送
- **路径追踪**：拦截 `cd` 命令自动更新当前目录，上传/下载相对路径自动拼接到 Shell 当前目录
- **PTY 按键缓冲**：二进制按键流中检测 `dl` 命令，防止漏给 bash

**6) 心跳保活线程**

```python
def _heartbeat(ws, reconnect_event):
    while not reconnect_event.is_set():
        ws.send("__PING__")
        time.sleep(30)
```

使用 `threading.Event` 实现线程间优雅退出。

**7) 三线程模型**

```
┌────────────────────────────────────────┐
│            Client 进程                  │
│                                        │
│ 主线程 → ws.recv() → PTY/管道 stdin    │
│ 读取线程 → PTY/管道 stdout → ws.send   │
│ 心跳线程 → __PING__ 每 30s             │
│                                        │
│ 文件传输 → 后台线程（不阻塞命令行）      │
└────────────────────────────────────────┘
```

### 2.4 认证协议设计

**握手流程**：

```
Backend:  IAM_BACKEND:<token>:<name>:<mode> ──► Relay
Frontend: AUTH:<token> 或 URL ?token=xxx ──► Relay → AUTH_OK
```

**协议细节**：

1. 所有连接必须在 30 秒内发送认证消息
2. 前端支持 URL 参数认证：`ws://host:port?token=<token>`
3. 不设 token 时保持向后兼容（不校验）
4. 认证失败返回 `AUTH_FAIL` + 断开（1008）
5. 重复后端名称拒绝注册（1008）

### 2.5 文件传输协议

上传流程：

```
F→B: __FILE_BEGIN:{b64path}:{size}    (上传请求)
B→F: __FILE_OK:{b64path}:{size}       (确认接收)
F→B: __FILE_CHUNK:{b64path}:{idx}:{b64}
F→B: __FILE_END:{b64path}
B→F: __FILE_DONE:{b64path}:{size}     (完成)
```

下载流程：

```
F→B: __FILE_DOWNLOAD:{b64path}         (请求)
B→F: __FILE_BEGIN:{b64path}:{size}     (开始)
B→F: __FILE_CHUNK:{b64path}:{idx}:{b64}
B→F: __FILE_END:{b64path}:{size}       (完成)
```

路径使用 base64 编码避免特殊字符冲突，分块大小 64KB。

### 2.6 CWD 追踪机制

Client 通过两种方式追踪 Shell 当前目录：

1. **文本通道**：Web 终端 JS 检测到 `cd <dir>` 命令时，额外发送 `__CWD:<dir>` 控制消息
2. **PTY 按键缓冲**：二进制按键流中缓冲字符，检测到完整的 `cd xxx` 命令后解析目录路径

追踪的 CWD 用于文件传输时拼接相对路径——用户不必输入完整绝对路径。

## 三、技术特色与工程亮点

### 3.1 异步/同步混合架构

**Relay（VPS 端）用异步**（`websockets`）：
- 需要同时处理多个前端和后端连接
- 异步 I/O 天然适合"多连接、低 CPU、高 I/O 等待"的场景

**Client（容器端）用同步**（`websocket-client`）：
- `websocket-client` 原生支持 `http_proxy_host` 参数
- 容器端只需一个连接，异步无优势
- 关键是：`websockets`（异步库）不原生支持 HTTP 代理隧道！

### 3.2 RelayState 类状态管理

与早期版本的闭包模式不同，当前版本使用 `RelayState` 类集中管理所有连接状态：

```python
class RelayState:
    backends: dict[str, Any]          # 后端名称→WS连接
    backend_modes: dict[str, str]     # 后端名称→模式(pty/pipe)
    backend_connected_at: dict[str, float]  # 连接时间戳
    frontends: set[Any]               # 前端WS连接
    frontend_targets: dict[Any, str|None]  # 前端→目标后端
    frontend_text_modes: dict[Any, bool]   # 前端→文本模式
```

这种设计比闭包更清晰：状态变量显式声明在 `__init__` 中，类型注解明确，方法签名自文档化。

### 3.3 防御性编程实践

**死连接清理**：每次广播前遍历前端集合，发送失败则标记为 dead，遍历结束后统一移除：

```python
dead = set()
for f in frontends:
    try:
        await f.send(payload)
    except Exception:
        dead.add(f)
frontends -= dead
```

**finally 清理保障**：任何退出路径都清理共享状态，不留僵尸连接。

**文件传输状态隔离**：用 `_file_transfers` 字典管理并发上传，各路径独立互不干扰。

### 3.4 向后兼容

- Token 可选：无 token 时不校验
- 命名可选：自动分配 `backend-N`
- 标签可选：单后端时不加 `[@name]`
- 模式可选：旧客户端不发送 mode 时默认为 `pipe`
- 认证方式可选：URL token / AUTH 消息 / 不认证

## 六、Hacker 技巧深度盘点

在软件工程与极客文化的语境下，wsstunnel 在面对看似无解的系统限制时，通过对底层协议、系统 API 和运行机制的深刻理解，运用了多个极具极客美学的巧妙技巧。

### 6.1 协议走私：HTTP CONNECT 降维打击

**问题**：目标容器仅允许 HTTP 出站，禁止任何裸 TCP 或 UDP 连接。

**技巧**：利用 `websocket-client` 库对 HTTP 代理的底层支持，发送 `CONNECT your-vps:443 HTTP/1.1`，把原本用于网页缓存的代理服务器变成一个**盲眼 TCP 隧道**，然后在隧道内完成 WebSocket `Upgrade` 握手。

```python
ws.connect(server_url,
    http_proxy_host=proxy_host,
    http_proxy_port=proxy_port,
)
```

**为什么高端**：防火墙以为在看网页，代理以为在建立 HTTPS 隧道，而实际上你在里面跑了一个拥有 root 权限的交互式 Shell。这在安全领域被称为**协议走私（Protocol Smuggling）**——用标准协议的"合理"行为绕过安全策略的检查。

### 6.2 本地 MitM：按键嗅探与指令劫持

**问题**：用户想在 Shell 里直接下载文件，但 bash 原生不支持下载命令。

**技巧**：Client 在 PTY 接收循环里实现了一个**按键嗅探器**，暗中监听每一个按键输入。当检测到用户输入 `dl <path>` 时，直接劫持这条指令：

```python
if ch in ("\r", "\n"):
    line = _key_buffer.replace("\r", "").replace("\n", "").strip()
    if line.startswith("dl "):
        # 劫持指令！不让 bash 执行
        os.write(master_fd, b"\x03\r")  # 塞入 Ctrl+C 清空 bash 输入
        threading.Thread(target=_send_file, args=(path, ws)).start()
```

**为什么高端**：这是典型的**中间人攻击（MitM）**思维。Client 不是简单的传声筒——它在用户的每个按键经过时都停下来看一眼。发现 `dl` 关键词后，它**替 bash 拦截了命令**，往 bash 里塞一个 Ctrl+C 清空行，然后自己启动后台线程传文件。用户感觉 bash 原生支持了下载，体验极度丝滑。

### 6.3 状态影子追踪：克隆 Shell 的内心世界

**问题**：父进程（Python）无法直接读取子进程（bash）的当前工作目录（CWD），但文件传输必须知道相对路径的基准。

**技巧**：在输入流中拦截并解析 `cd` 命令，在 Python 内存里维护一份 CWD 的"影子副本"：

```python
def _update_cwd(cmd):
    if cmd.startswith("cd "):
        target = parts[1]
        if not os.path.isabs(target):
            target = os.path.normpath(os.path.join(_cwd, target))
        _cwd = target
```

同时，Web 终端 JS 在检测到 `cd` 命令时，额外发送 `__CWD:<dir>` 控制消息，即使在后端按键缓冲模式下也能同步 CWD。

**为什么高端**：子进程的状态对父进程是透明隔离的，无法直接读取。你通过在数据流中**「克隆」一份 Shell 的状态机**，实现了对隔离状态的旁路追踪。这种影子追踪技术常用于高级安全沙箱和系统调用拦截。

### 6.4 内核级欺骗：虚拟 TTY 的艺术

**问题**：普通管道模式下，bash 检测到自己没有连接真实终端，会禁用行编辑、禁用 TUI 程序，`vim` 和 `top` 直接崩溃。

**技巧**：调用 `pty.openpty()` 创建一对伪终端，让 bash 以为面前坐着一个拿着物理键盘的人：

```python
master_fd, slave_fd = pty.openpty()
_set_winsize(master_fd, rows, cols)
shell_proc = subprocess.Popen(
    [shell, "-i"],
    stdin=slave_fd, stdout=slave_fd, stderr=slave_fd,
    preexec_fn=os.setsid,  # 创建新进程组
)
```

配合 `__RESIZE:rows,cols` 调用 `fcntl.ioctl(fd, termios.TIOCSWINSZ)`，不仅骗过内核，还接管了终端控制器的底层信号。

**为什么高端**：`preexec_fn=os.setsid` 让 bash 成为新会话的组长——内核会以为这个进程组连接着真正的终端。当 bash 收到 `SIGWINCH`（窗口变化信号）时，它以为显示器在调整尺寸。普通反向 Shell 跑 `vim` 会直接崩溃，而这里 `vim` 能全屏运行，因为内核被彻底欺骗了。

### 6.5 协议信道复用：一根线上跑多个服务

**问题**：需要同时传输 Shell 画面、控制信号、文件数据，不能开多个连接。

**技巧**：利用 WebSocket 原生的帧类型机制，在单连接上实现多路复用：

```python
# 二进制帧 → 纯净的 PTY 原始画面流
if isinstance(message, bytes):
    await _forward_binary_to_frontends(...)

# 文本帧 → 控制信令 + 文件数据
if isinstance(message, str) and message.startswith("__FILE_"):
    await _forward_to_frontends_untagged(...)
```

**为什么高端**：没有引入任何额外的序列化开销（protobuf/JSON），直接通过 `isinstance(msg, bytes)` 就在一根 TCP 连接上实现了控制面（Control Plane）和数据面（Data Plane）的完美隔离。文件块、心跳、PTY 画面、窗口调整、信号转发——五种流量在一根线上有序共存，互不干扰。

### 6.6 DOM 层事件劫持：给黑盒打热补丁

**问题**：移动端通过 xterm.js 输入中文时，键盘事件被 IME 组合输入吞掉，导致漏字、乱码。

**技巧**：绕过 xterm.js 的顶层 API，直接潜入底层 DOM 事件 `beforeinput`，并配合去重锁：

```javascript
term.textarea.addEventListener('beforeinput', (e) => {
    if (e.inputType === 'insertLineBreak') {
        mobileSend('\n');
    } else if (e.data && (e.inputType === 'insertText' || e.inputType === 'insertFromComposition')) {
        mobileSend(e.data);
    }
});
```

**为什么高端**：xterm.js 是重型第三方库，其事件封装是黑盒。当黑盒有缺陷时，真正深入底层——在浏览器原生 API 层通过事件劫持 + 防抖锁，从外部给黑盒打上完美热补丁。这体现了 tap into the lower layer 的极客精神。

### 6.7 总结

在这套代码中，wsstunnel 扮演了多个维度的「骗子」：

| 维度 | 骗过了谁 | 用了什么 |
|------|---------|---------|
| 网络 | 防火墙/代理 | HTTP CONNECT 协议走私 |
| 内核 | Linux 进程管理 | `pty.openpty()` + `os.setsid` |
| 用户 | 命令行操作者 | `_key_buffer` 按键嗅探 |
| 子进程 | bash Shell | `_cwd` 状态影子追踪 |
| 浏览器 | xterm.js 黑盒 | `beforeinput` 事件劫持 |

这不是业务 CRUD，而是真正与网络协议栈、Linux 内核和浏览器 DOM 斗智斗勇的工程实践。

## 七、已知限制

| 限制 | 说明 |
|------|------|
| 消息时序 | 管道模式下多条命令输出可能交错。PTY 模式已解决 |
| 无压缩 | 大量输出效率不高 |
| 仅限 Shell | 无法转发其他 TCP 服务 |
| 无审计日志 | 缺少结构化的命令记录 |
| CWD 追踪不完美 | 仅追踪显式 `cd` 命令，无法感知程序内部 `chdir` |
