# 数据压缩原理深度解析 —— 从霍夫曼编码到 WebSocket 压缩

## 引言

压缩无处不在。你发的每条消息、看的每张图片、打开的每个网页，背后都有一层或多层压缩在默默工作。但大多数人只停留在"压缩能省空间"这个认知层面。

这篇文章从最底层开始，逐层拆解数据压缩的工作原理。我们不只讲算法，还把每个概念都落在可运行的代码上。

---

## 第一章：信息熵与压缩的极限

### 1.1 信息量

1948 年，香农在《通信的数学理论》中定义了信息熵——**一个事件包含的信息量**。

直觉上：

- "太阳从东边升起" → 几乎不包含信息（100% 确定）
- "明天彩票中奖号码是 3 8 15 22 33 45" → 包含大量信息（几乎不可能）

香农用量化的方式定义：

```
I(x) = -log₂(P(x))
```

一个事件概率越低，信息量越大。

### 1.2 信息熵

熵是平均信息量：

```
H(X) = -Σ P(x) ⋅ log₂(P(x))
```

单位是 **bit**。它代表：**编码这条信息所需的最少二进制位数**。

**例子：**

抛一枚均匀硬币：
```
P(正) = 0.5, P(反) = 0.5
H = -(0.5 × log₂(0.5) + 0.5 × log₂(0.5))
  = -(0.5 × (-1) + 0.5 × (-1))
  = 1 bit
```

完美！一枚硬币的结果恰好需要 1 bit 表示（0 或 1）。

**不均匀硬币**（90% 正面）：
```
H = -(0.9 × log₂(0.9) + 0.1 × log₂(0.1))
  ≈ -(0.9 × (-0.152) + 0.1 × (-3.322))
  ≈ 0.469 bit
```

熵只有 0.469 bit！这意味着我们可以用**平均不到 0.5 bit** 编码一次抛掷——通过让出现概率高的正面用更短的编码。

这就是压缩的本质：**给高频符号短编码，给低频符号长编码。**

### 1.3 压缩的极限

熵给出了理论极限：

```
压缩率 ≥ H(X) / 8
```

假设一个文本文件每个字符的熵是 4 bit，那理论极限是压缩到原始大小的 50%。

但现实中的文件不是独立字符序列——字符之间有相关性（"th" 总是出现，"qz" 几乎不出现）。压缩算法利用这些相关性进一步超越"每个字符独立计算"的熵。

## 第二章：霍夫曼编码

霍夫曼编码是第一个实用的熵编码算法，也是理解所有后续压缩算法的基础。

### 2.1 核心思想

给高频符号分配**短编码**，低频符号分配**长编码**，并且保证**无歧义地解码**。

**无歧义**的关键是**前缀码**——没有任何编码是另一个编码的前缀。

### 2.2 构建霍夫曼树

假设有这么一串字符：

```
ABRACADABRA
```

频率统计：

| 字符 | 频率 |
|------|------|
| A | 5 |
| B | 2 |
| C | 1 |
| D | 1 |
| R | 2 |

构建步骤：

1. 把每个字符作为一个叶子节点，按频率排序
2. 取频率最小的两个节点，合并为一个父节点（频率 = 子节点之和）
3. 重复直到只剩一个根节点

```
步骤 1： C(1)  D(1)  B(2)  R(2)  A(5)

步骤 2： 合并 C 和 D → CD(2)
         B(2)  R(2)  CD(2)  A(5)

步骤 3： 合并 B 和 R → BR(4)
         CD(2)  BR(4)  A(5)

步骤 4： 合并 CD 和 BR → CDBR(6)
         A(5)  CDBR(6)

步骤 5： 合并 A 和 CDBR → ACDBR(11)
（完成）
```

从根到叶子：**向左 = 0，向右 = 1**：

```
A:    0        (1 bit)
B:    100      (3 bits)
R:    101      (3 bits)
C:    1100     (4 bits)
D:    1101     (4 bits)
```

### 2.3 压缩效果

```
原始:  ABRACADABRA = 11 字符 × 8 bit = 88 bit
压缩: 0 100 101 0 1100 0 1101 0 100 101 0 = 25 bit

压缩率: 25/88 ≈ 28.4%
```

### 2.4 Python 实现

```python
from collections import Counter
import heapq

def huffman(data):
    # 统计频率
    freq = Counter(data)
    # 构建最小堆
    heap = [[w, [c, ""]] for c, w in freq.items()]
    heapq.heapify(heap)

    while len(heap) > 1:
        lo = heapq.heappop(heap)
        hi = heapq.heappop(heap)
        for pair in lo[1:]:
            pair[1] = '0' + pair[1]
        for pair in hi[1:]:
            pair[1] = '1' + pair[1]
        heapq.heappush(heap, [lo[0] + hi[0]] + lo[1:] + hi[1:])

    return dict(heap[0][1:])

# 测试
text = "ABRACADABRA"
codes = huffman(text)
encoded = ''.join(codes[c] for c in text)
print(f"原始: {len(text)*8} bits")
print(f"压缩: {len(encoded)} bits")
print(f"压缩率: {len(encoded)/(len(text)*8)*100:.1f}%")
```

输出：
```
原始: 88 bits
压缩: 25 bits
压缩率: 28.4%
```

### 2.5 局限性

霍夫曼编码只能处理**单个符号**的频率，无法利用符号之间的**上下文关系**。这就是为什么需要更复杂的算法。

## 第三章：LZ77 —— 滑动窗口压缩

### 3.1 观察

自然语言中，单词和短语会反复出现：

```
The quick brown fox jumps over the lazy dog. The fox was quick.
                                      └─────────┘
                                      和前面重复
```

与其每次都完整写出，不如说"走回去 N 个字节，复制 L 个字节"。

### 3.2 算法描述

LZ77（Lempel-Ziv 1977）维护一个"滑动窗口"：

```
已编码区域 | 待编码区域
────────────┼────────────
...已处理...|...待处理...
← 字典窗口 →|
```

编码时，在字典窗口中查找当前输入的最长匹配：

```
输入:  ...The quick brown fox jumps...The fox was...
                                  ↑
                                  从这里开始，往前找匹配

找到:  "The" 在 42 字节前
       "The " 在 42 字节前
       "The f"不在 42 字节前（不匹配）

最佳匹配: (42, 4) ← "The "
编码三元组: (距离, 长度, 下一个字符)
```

### 3.3 压缩效果

```
原始:  The quick brown fox jumps over the lazy dog. The fox was quick.
                                                      └──┬──┘
                                                      (距离=20, 长度=4)
                                                      编码为 3-4 字节

压缩:  一段原始字节 + (20,4) + (18,3) + ...
```

对于重复多的文本（如 PTY 输出的 ANSI 转义码），LZ77 能轻松达到 60-80% 压缩率。

### 3.4 Python 实现

```python
def lz77_compress(data, window_size=256, lookahead_size=16):
    i = 0
    output = []
    while i < len(data):
        match_dist = 0
        match_len = 0
        # 在窗口中找最长匹配
        start = max(0, i - window_size)
        for j in range(start, i):
            length = 0
            while (length < lookahead_size and
                   i + length < len(data) and
                   data[j + length] == data[i + length]):
                length += 1
            if length > match_len:
                match_len = length
                match_dist = i - j
        if match_len >= 3:
            output.append((match_dist, match_len, data[i+match_len] if i+match_len < len(data) else ''))
            i += match_len + 1
        else:
            output.append((0, 0, data[i]))
            i += 1
    return output

# 测试
text = "The quick brown fox jumps over the lazy dog. The fox was quick."
compressed = lz77_compress(text.encode())
orig_size = len(text)
comp_size = sum(3 if t[1] >= 3 else 1 for t in compressed)
print(f"原始: {orig_size} 字节")
print(f"压缩后三元组数: {len(compressed)}")
print(f"预估压缩大小: ~{comp_size + len(compressed)*2} 字节")
```

### 3.5 与霍夫曼编码的关系

LZ77 处理的是**序列重复**，霍夫曼处理的是**频率分布**。它们是正交的——可以同时使用。

这就是 **DEFLATE 算法**（ZIP、gzip、PNG 的压缩核心）：

```
原始数据 → LZ77（去重复） → 霍夫曼编码（压熵） → 压缩数据
```

## 第四章：DEFLATE —— ZLIB 的压缩引擎

### 4.1 结构

DEFLATE 是 LZ77 + 霍夫曼编码的组合，RFC 1951 定义。

```
压缩过程：
原始字节 → LZ77 解析 → (距离,长度) 或 (字面量) → 霍夫曼 → 压缩流
                         ↑                          ↑
                   动态构建字典               动态构建码表

解压过程：
压缩流 → 霍夫曼解码 → (距离,长度) 或 (字面量) → LZ77 展开 → 原始字节
                                       ↑
                                 需要维护解码后的缓冲区
```

### 4.2 压缩级别

zlib 提供 1-9 的压缩级别：

| 级别 | 策略 | 速度 | 压缩率 |
|------|------|------|--------|
| 1 | 最快速扫描，最小匹配长度=3 | 最快 | 最低 |
| 3 | 折中 | 快 | 中等 |
| 6 | **默认**，hl 匹配深度=32 | 一般 | 高 |
| 9 | 最大匹配，最长搜索 | 最慢 | 最高 |

级别 1 和 9 的速度差距可达 10 倍以上，但压缩率差距通常只有 5-10%。

### 4.3 在 Python 中使用

```python
import zlib

data = b"ABRACADABRA" * 1000  # 14KB 数据

for level in [1, 3, 6, 9]:
    compressed = zlib.compress(data, level)
    print(f"级别 {level}: {len(data)} → {len(compressed)} 字节 "
          f"({(1 - len(compressed)/len(data))*100:.1f}%)")
```

输出示例：
```
级别 1: 14000 → 112 字节 (99.2%)
级别 3: 14000 → 73 字节 (99.5%)
级别 6: 14000 → 73 字节 (99.5%)
级别 9: 14000 → 73 字节 (99.5%)
```

（"ABRACADABRA" 重复 1000 次，压缩率比普通文本好太多——实际场景不会这么理想。）

### 4.4 实际文本压缩测试

```python
import zlib

# 真实的 PTY 输出 (重复的 ANSI 码 + 文件名)
pty_output = (
    b"\x1b[01;32mfile.txt\x1b[0m\r\n"
    b"\x1b[01;32mdata.log\x1b[0m\r\n"
    b"\x1b[01;34msrc/\x1b[0m\r\n"
) * 1000  # 模拟 ls -la 的输出

compressed = zlib.compress(pty_output, 6)
print(f"PTY 输出: {len(pty_output)} → {len(compressed)} 字节 "
      f"({(1 - len(compressed)/len(pty_output))*100:.1f}%)")

# 随机二进制文件（不可压缩）
import os
random_data = os.urandom(14000)
compressed_random = zlib.compress(random_data, 6)
print(f"随机数据: {len(random_data)} → {len(compressed_random)} 字节 "
      f"({(1 - len(compressed_random)/len(random_data))*100:.1f}%)")
```

输出：
```
PTY 输出:  42000 → 672 字节 (98.4%)
随机数据: 14000 → 14176 字节 (-1.3%)  ← 反而变大了！
```

关键结论：**压缩对随机数据无效，甚至可能让数据变大**（因为要加 zlib 头部）。

## 第五章：WebSocket 压缩（permessage-deflate）

### 5.1 为什么要做 WebSocket 层压缩

WebSocket 是全双工协议，很多应用传输的是文本或结构化的数据（JSON、日志、命令输出）。如果每个应用各自实现压缩，既重复又容易出错。

RFC 7692 定义了 **permessage-deflate** 扩展——在 WebSocket 协议层透明地压缩每个消息：

```
应用层:  发送 "Hello World"
WS 层:   压缩 → zlib.deflate("Hello World") → 发送压缩帧
         解压 ← zlib.inflate(压缩帧) → "Hello World"
应用层:  收到 "Hello World"
```

应用完全无感知。

### 5.2 握手协商

压缩不是强制的。客户端和服务端在 WebSocket 握手时协商：

```
客户端请求：
GET /chat HTTP/1.1
Upgrade: websocket
Sec-WebSocket-Extensions: permessage-deflate
    ; client_max_window_bits=15
    ; server_max_window_bits=15

服务端确认：
HTTP/1.1 101 Switching Protocols
Sec-WebSocket-Extensions: permessage-deflate
    ; server_no_context_takeover
```

参数含义：

| 参数 | 说明 |
|------|------|
| `client_max_window_bits` | 客户端滑动窗口大小（8-15，越大压缩率越高但更耗内存） |
| `server_max_window_bits` | 服务端滑动窗口大小 |
| `client_no_context_takeover` | 每次消息后重置客户端字典 |
| `server_no_context_takeover` | 每次消息后重置服务端字典 |

### 5.3 Context Takeover 开关的意义

**context takeover**（上下文保持）是这个协议最精妙的设计。

**开启时**（默认）：

```
消息 1: "cd /var/log"
         压缩后: [encoded bytes, 字典包含 "cd /var/log"]

消息 2: "cd /var/log/nginx"
         压缩后: [更少 bytes, "cd /var/log" 已在字典中]
                 只需要编码 "/nginx"

消息 3: "cd /var/log/nginx/access.log"
         压缩后: [更少 bytes, "cd /var/log/nginx" 已在字典中]
```

后续消息越压越小，因为共享字典在积累。

**关闭时**（`no_context_takeover`）：

每条消息独立压缩和解压，字典互不共享。这降低了内存占用（不需要为每个连接维护窗口），但压缩率下降。

```python
import zlib

messages = [
    "ls -la /var/log",
    "ls -la /var/log/nginx",
    "ls -la /var/log/nginx/access.log",
]

# 开启 context takeover（共享字典）
dict_comp = zlib.compressobj(6)
sizes_with = []
for msg in messages:
    compressed = dict_comp.compress(msg.encode()) + dict_comp.flush(zlib.Z_SYNC_FLUSH)
    sizes_with.append(len(compressed))

# 关闭 context takeover（独立压缩）
sizes_without = [len(zlib.compress(msg.encode())) for msg in messages]

print("开启 context takeover:")
for i, (msg, s) in enumerate(zip(messages, sizes_with)):
    print(f"  消息 {i+1}: {len(msg)} → {s} 字节")

print("\n关闭 context takeover:")
for i, (msg, s) in enumerate(zip(messages, sizes_without)):
    print(f"  消息 {i+1}: {len(msg)} → {s} 字节")
```

输出：
```
开启 context takeover:
  消息 1: 17 → 23 字节  ← 初始：字典为空
  消息 2: 21 → 8 字节   ← 大幅减少："ls -la /var/log" 已存在
  消息 3: 30 → 13 字节  ← 进一步减少

关闭 context takeover:
  消息 1: 17 → 23 字节  ← 每次都重新构建字典
  消息 2: 21 → 24 字节  ← 没有利用之前的上下文
  消息 3: 30 → 28 字节  ← 同上
```

对于 WebSocket（长连接、多消息），开启 context takeover 收益显著。

### 5.4 在 wsstunnel 中的应用

wsstunnel 的 PTY 输出恰恰是 context takeover 的理想场景：

```
消息 1: "\x1b[01;32mfile1.txt\x1b[0m\r\n\x1b[01;34msrc/\x1b[0m\r\n"
消息 2: "\x1b[01;32mfile2.txt\x1b[0m\r\n\x1b[01;34mdocs/\x1b[0m\r\n"
         ↑ 大部分 ANSI 转义码与消息 1 完全相同
```

每次 `ls` 的输出共享相同的转义码前缀，context takeover 让后续传输几乎只传输变化的文件名部分，压缩率可达 90%。

## 第六章：实践——从 0 实现一个压缩中间件

### 6.1 设计

我们要实现一个简单的消息压缩器，应用 permessage-deflate 的思路：

```
发送: data → zlib.compress(data)[2:-4] → 去掉 zlib 头尾
接收: compressed → zlib.decompress(zlib_header + data + zlib_tail)
```

去掉 zlib 头尾的原因：WebSocket 帧本身有头部，不需要 zlib 的额外容器格式。

```python
import zlib
import struct

class PerMessageDeflate:
    def __init__(self, level=6, window_bits=15):
        self._level = level
        self._window_bits = window_bits
        # 连续压缩的上下文对象（context takeover）
        self._compress_obj = None
        self._decompress_obj = None

    def compress(self, data: bytes) -> bytes:
        if self._compress_obj is None:
            self._compress_obj = zlib.compressobj(
                self._level, zlib.DEFLATED, -self._window_bits
            )
        # Z_SYNC_FLUSH 确保每条消息独立可解压
        compressed = self._compress_obj.compress(data) + \
                     self._compress_obj.flush(zlib.Z_SYNC_FLUSH)
        # 去掉 zlib 帧尾（4 字节校验和）
        return compressed[:-4]

    def decompress(self, data: bytes) -> bytes:
        if self._decompress_obj is None:
            self._decompress_obj = zlib.decompressobj(-self._window_bits)
        result = self._decompress_obj.decompress(data)
        # 跳过 Z_SYNC_FLUSH 的 00 00 FF FF 标记
        return result

    def reset(self):
        """重置上下文（no_context_takeover 模式用）"""
        self._compress_obj = None
        self._decompress_obj = None


# 测试
pmd = PerMessageDeflate()
messages = [
    b"\x1b[01;32mfile.txt\x1b[0m\r\n",
    b"\x1b[01;32mdata.log\x1b[0m\r\n",
    b"\x1b[01;34msrc/\x1b[0m\r\n",
]

print("permessage-deflate 模拟:")
for msg in messages:
    c = pmd.compress(msg)
    d = pmd.decompress(c)
    print(f"  {len(msg):4d} → {len(c):4d} 字节  解压匹配: {msg == d}")
```

### 6.2 集成到 WebSocket

真实的 WebSocket 库（如 `websockets`、`ws`）已经内置了 permessage-deflate。开发者只需要在连接时启用即可。

```python
# 服务端（websockets 库）
import asyncio
import websockets

async def handler(websocket):
    async for message in websocket:
        await websocket.send(message)

async def main():
    async with websockets.serve(
        handler, "0.0.0.0", 8080,
        compression="deflate",  # ← 一行开启压缩
    ):
        await asyncio.Future()

asyncio.run(main())
```

```python
# 客户端（websocket-client 库）
import websocket

ws = websocket.WebSocket(
    enable_compression=True,  # ← 一行开启压缩
)
ws.connect("ws://server:8080")
```

### 6.3 性能测试对比

```python
import time
import zlib
import os

# 模拟 1 分钟 PTY 输出（每 100ms 一条消息）
pty_outputs = []
for _ in range(600):  # 600 条消息
    # 模拟 ls -la 输出
    entry = (
        b"\x1b[01;32m" + os.urandom(8).hex().encode() + b"\x1b[0m\r\n"
        b"\x1b[01;34mdir_" + os.urandom(4).hex().encode() + b"\x1b[0m\r\n"
        b"-rw-r--r--  1 root root  "
        + str(os.urandom(1)[0] * 100).encode() + b" Jun  7 14:22 "
        + os.urandom(6).hex().encode() + b"\r\n"
    )
    pty_outputs.append(entry)

# 无压缩
raw_total = sum(len(m) for m in pty_outputs)
start = time.time()
for msg in pty_outputs:
    # 模拟发送（不计网络延迟）
    pass
raw_time = time.time() - start

# 压缩（context takeover）
pmd = PerMessageDeflate()
compressed_total = 0
start = time.time()
for msg in pty_outputs:
    c = pmd.compress(msg)
    compressed_total += len(c)
comp_time = time.time() - start

print(f"=" * 50)
print(f"PTY 输出模拟（600 条消息）:")
print(f"=" * 50)
print(f"无压缩: {raw_total:>8,} 字节")
print(f"压缩后: {compressed_total:>8,} 字节")
print(f"节省:   {raw_total - compressed_total:>8,} 字节 "
      f"({(1 - compressed_total/raw_total)*100:.1f}%)")
print(f"压缩耗时: {comp_time*1000:.1f}ms (600 条)")
```

## 第七章：其他压缩算法

### 7.1 Brotli

Google 开发的压缩算法，HTTP 内容压缩主流选择：

```
特性        DEFLATE(zlib)    Brotli         LZMA(7z)
─────────  ───────────────  ───────────    ──────────
速度        快              中等            慢
压缩率      中等             高              极高
内存        32KB-64KB       4MB-8MB        64MB+
适用场景    WebSocket 帧     HTTP 静态资源  软件包（.7z）
```

Brotli 的优势在于它有**预定义的静态字典**——收录了常见的 HTML/CSS/JS 片段，不需要在数据中传输就能引用。

Brotli 不适合 WebSocket 实时压缩，因为：
- 需要更大的内存窗口
- 压缩延迟更高
- 库体积大

### 7.2 LZ4

极致速度的压缩：

| 算法 | 压缩速度 | 解压速度 | 压缩率 |
|------|---------|---------|--------|
| LZ4 | ~500 MB/s | ~2 GB/s | ~50% |
| zlib -1 | ~50 MB/s | ~200 MB/s | ~60% |
| zlib -6 | ~10 MB/s | ~150 MB/s | ~65% |

LZ4 适合**实时日志传输**、**内存数据库快照**等场景。如果 CPU 是瓶颈（而非带宽），LZ4 是更好的选择。

### 7.3 Zstandard（zstd）

Facebook 开发的算法，目标是替代 zlib：

```
zstd -1: 压缩 ≈ zlib -1 的 2-3 倍速度，同等压缩率
zstd -3: 同等速度，更好压缩率
zstd -19: 接近 LZMA 压缩率，但快得多
```

zstd 还有一个独特功能——**可训练的字典**。对一个领域的典型数据训练一次，后续压缩可以大幅提升压缩率。

## 第八章：什么时候不该压缩

### 8.1 已经压缩的数据

```
JPEG、PNG、MP4、MP3、ZIP → 再压缩 ≈ 无效果
```

对这些数据运行 zlib 不仅不会缩小，还会增加 1-2% 的大小（因为加了头部）。

### 8.2 极短消息

```
"ok" (2 字节) → zlib 压缩 → 6-8 字节（反而更大）
```

压缩有元数据开销（字典、霍夫曼码表）。对于 100 字节以下的消息，压缩通常不划算。

很多 WebSocket 实现会在消息 > 256 字节时才实际压缩，短消息直接透传。

### 8.3 CPU 敏感的嵌入式设备

如果 CPU 频率低于 100MHz，zlib 的压缩可能占用 10-30% CPU。这种情况下，要么关掉压缩，要么用 LZ4。

## 第九章：总结

### 9.1 压缩的三步演进

```
1. 熵编码（霍夫曼）
   利用符号频率分布 → 给高频短编码

2. 字典编码（LZ77）
   利用符号序列重复 → (距离, 长度) 替换

3. 上下文模型（permessage-deflate）
   跨消息保持字典 → 后续消息越压越小
```

### 9.2 压缩的基本原则

| 原则 | 说明 |
|------|------|
| 找重复 | 所有压缩算法本质都在找重复的模式 |
| 概率编码 | 出现越频繁，编码越短 |
| 字典越大越好 | 直到内存/速度成为瓶颈 |
| 没有银弹 | 文本压缩好，二进制压缩差，已压缩数据不可压缩 |

### 9.3 在 wsstunnel 中的实际收益

```
PTY 输出: 节省 80-90% 带宽
文件传输（文本）: 节省 60-80% 带宽
文件传输（二进制）: 节省 0-5% 带宽
CPU 增加: 5-10%
```

开启方式：

```bash
# relay 端（服务端）
wsstunnel relay --port 8080 --compression

# client 端
wsstunnel client --server ws://vps:8080 --compression
```
