# ws-tunnel 深度解析：从零到一的 WebSocket 远程 Shell 中继架构

> 一款以 536 行 Python 代码构建的生产级远程 Shell 隧道工具，具备 HTTP 代理穿透、多后端路由、TLS 加密、心跳保活与指数退避重连能力。

---

## 一、项目概述与定位

### 1.1 项目定义

**ws-tunnel**（PyPI 包名 `wsstunnel`）是一个基于 WebSocket 协议的远程 Shell 中继工具，专为穿透受限网络环境而设计。它的核心能力是：在目标设备仅能发起 HTTP 出站连接的极端网络条件下，仍能建立一条稳定的双向交互式 Shell 通道。

这个项目解决了一个非常具体的痛点——当你面对一个严格受限的容器环境（如在线 IDE 沙箱、CI/CD runner、受限办公网络中的设备），该环境仅允许通过 HTTP 代理访问外部网络，不允许任何直接的 TCP 连接，也不允许 UDP、ICMP 等协议出站。在这样严苛的条件下，传统的 SSH、frp、ngrok 等方案统统失效，而 ws-tunnel 通过 HTTP CONNECT 代理隧道 + WebSocket 协议伪装的组合拳，优雅地突破了这一限制。

## 二、系统架构深度剖析

### 2.1 三方架构模型

ws-tunnel 采用了经典的三方中继（Relay）架构，这也是 NAT 穿透和内网穿透领域最成熟的设计模式之一。三个参与方各有明确的职责边界：

```
┌─────────────────────────────────────────────────────────────────┐
│                        三方架构模型                              │
│                                                                 │
│   Frontend（前端）          Relay（中继）        Backend（后端）  │
│   ┌──────────┐           ┌──────────┐         ┌──────────┐     │
│   │ 第三方    │           │ VPS      │         │ 目标容器  │     │
│   │ 操作者    │◄─WebSocket─►│ 中继服务 │◄─WebSocket─►│ 受限设备  │     │
│   │          │           │          │         │          │     │
│   │ 发送命令  │           │ 路由转发  │         │ 执行命令  │     │
│   │ 接收输出  │           │ 身份验证  │         │ 返回结果  │     │
│   └──────────┘           └──────────┘         └──────────┘     │
│       直连                    公网                  HTTP代理     │
└─────────────────────────────────────────────────────────────────┘
```

这种架构的设计精髓在于：**所有连接都由客户端主动发起**。Backend（位于受限网络中的目标设备）主动向 Relay（部署在公网 VPS 上的中继服务）建立 WebSocket 连接，绕过了 NAT 和防火墙的入站限制。Frontend（操作者）同样主动连接 Relay，通过 Relay 的路由机制与 Backend 间接通信。

### 2.2 为什么选择中继模式而非 P2P？

在远程控制领域，存在两种主要架构模式：

**P2P 直连模式**（如 chisel 的某些配置）：两端尝试直接建立连接，通常需要 STUN/TURN 服务器辅助 NAT 打洞。优点是延迟低，缺点是在对称 NAT 和严格防火墙环境下成功率低，且实现复杂度高。

**中继模式**（ws-tunnel 的选择）：所有流量经过一个中间节点转发。虽然增加了一跳延迟，但连接成功率接近 100%，且实现简洁。对于远程 Shell 这种交互式场景，几十毫秒的额外延迟完全可以接受。

ws-tunnel 选择中继模式是深思熟虑的结果：目标设备处于极端受限网络中，P2P 打洞几乎不可能成功（HTTP 代理只允许 HTTP CONNECT 方法，不支持 UDP 和裸 TCP），而中继模式下 Backend 只需发起一个看似普通 HTTP 请求的 WebSocket 连接，巧妙地利用了代理服务器的 CONNECT 隧道能力。

### 2.3 组件职责详解

#### 2.3.1 Relay（中继服务）—— `relay.py`

Relay 是整个系统的中枢神经，承担以下核心职责：

**1) 连接管理**

Relay 使用 `websockets` 异步库（基于 Python asyncio）监听 WebSocket 连接。每个新连接进入后，首先等待第一条消息（30 秒超时），根据消息内容判断连接身份：

```python
first = await asyncio.wait_for(websocket.recv(), timeout=30)
backend_info = _parse_backend_auth(first, token)
if backend_info:
    # 注册为后端
elif _is_frontend_auth(first, token):
    # 注册为前端
else:
    # 认证失败，断开
```

这种"首条消息决定角色"的设计极为精简——无需复杂的握手协议、无需额外的 API 端点、无需证书区分，一条消息搞定身份认证和角色注册。

**2) 后端注册与命名**

后端连接通过发送 `IAM_BACKEND:<token>:<name>` 格式的消息完成注册。系统支持两种命名方式：

- **自动命名**：仅发送 `IAM_BACKEND:<token>`，系统自动分配 `backend-1`、`backend-2` 等名称
- **自定义命名**：发送 `IAM_BACKEND:<token>:my-container`，使用指定名称

注册后立即广播通知所有前端后端列表变化：

```python
backends[name] = websocket
await _broadcast_backend_list(frontends, backends)
```

**3) 前端命令路由**

前端消息处理是 Relay 的智能路由核心，支持三种路由模式：

| 路由模式 | 消息格式 | 行为 |
|---------|---------|------|
| 粘性切换 | `USE <name>` | 设置当前前端的默认目标后端，后续命令直接发送 |
| 查看当前 | `USE` | 显示当前使用的后端名称（或 auto 模式） |
| 命名路由 | `@name <cmd>` | 临时发送给指定名称的后端（不影响当前选择） |
| 列表查询 | `LIST` | 返回所有已连接后端的名称、模式和当前选择标记 |
| 窗口调整 | `__RESIZE:rows,cols` | 调整 PTY 窗口大小 |
| 信号转发 | `__SIGNAL:SIGXXX` | 向后端进程组发送信号（如 SIGINT） |
| 默认路由 | `<cmd>` | 发送给当前目标后端（USE 选中的 / 第一个注册的） |

这种设计兼顾了向后兼容性和多后端灵活性。旧版前端（不知道 `@name` 语法）仍可通过默认路由正常工作，新版前端则可以精确控制目标后端。

**4) 输出广播与标签**

当多个后端同时存在时，后端输出会自动添加 `[@name]` 标签，以便前端区分来源：

```python
await _forward_to_frontends(
    frontends, message, name if len(backends) > 1 else None
)
```

当只有一个后端时，输出保持原样（无标签），保证向后兼容。

**5) 心跳处理**

Relay 响应后端发来的 `__PING__` 心跳消息，返回 `__PONG__`：

```python
if message == "__PING__":
    await websocket.send("__PONG__")
    continue
```

同时在 WebSocket 协议层面，配置了 20 秒 ping 间隔和 10 秒超时：

```python
websockets.serve(handler, host, port,
    ping_interval=20,
    ping_timeout=10,
)
```

这形成了**双层心跳机制**：WebSocket 协议级别的 ping/pong 帧用于检测连接活性，应用层面的 `__PING__`/`__PONG__` 用于检测端到端可达性。

**6) TLS 加密**

Relay 支持通过 SSL 证书启用 wss:// 加密：

```python
def _create_ssl_context(cert_path, key_path):
    ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
    ctx.load_cert_chain(cert_path, key_path)
    return ctx
```

兼容三种 TLS 部署模式：
- 直接加载证书文件（适合 Let's Encrypt）
- 自签名证书（配合客户端 `--insecure` 参数）
- nginx 反向代理（推荐生产环境，证书管理交给 nginx）

**7) 闭包式状态管理**

Relay 的状态管理采用了 Python 闭包模式，这是一个值得品味的设计选择：

```python
def _make_handler(token):
    backends: dict[str, object] = {}  # 闭包捕获的状态
    frontends: set = set()

    async def handler(websocket):
        nonlocal backends, frontends
        # ... 处理逻辑 ...

    return handler
```

使用闭包而非类来管理状态，好处是代码更紧凑，所有状态变量的生命周期与 handler 绑定，不需要额外的上下文对象。`backends` 和 `frontends` 集合在 `_make_handler` 调用时创建，被所有 WebSocket 连接共享，形成了一个轻量级的连接池。

#### 2.3.2 Client（后端客户端）—— `client.py`

Client 是部署在目标设备上的轻量级代理，承担以下职责：

**1) HTTP 代理穿透**

这是 ws-tunnel 最核心的差异化能力。Client 使用 `websocket-client` 同步库（而非异步的 `websockets`），原因在于它原生支持 HTTP CONNECT 代理：

```python
ws.connect(
    server_url,
    http_proxy_host=proxy_host,
    http_proxy_port=proxy_port,
)
```

当指定代理参数时，`websocket-client` 库会先向代理服务器发送 HTTP CONNECT 请求：

```
CONNECT target-vps:8080 HTTP/1.1
Host: target-vps:8080
```

代理服务器收到请求后，建立一条到目标 VPS 的 TCP 隧道，此后所有 WebSocket 帧都通过这条隧道传输。从网络防火墙的角度看，这只是一次普通的 HTTPS 代理请求，完全符合安全策略。

**2) 指数退避重连**

网络环境不稳定是受限容器环境的常态。Client 实现了指数退避重连策略：

```python
attempt = 0
while True:
    try:
        # 连接 + 工作...
        attempt = 0  # 成功后重置
    except Exception:
        attempt += 1
        delay = min(
            reconnect_interval * (2 ** (attempt - 1)),
            _RECONNECT_MAX_DELAY,  # 300 秒封顶
        )
        time.sleep(delay)
```

重连间隔从 5 秒开始，每次翻倍：5s → 10s → 20s → 40s → ... → 300s（封顶）。这避免了在服务器故障时的高频重试风暴，同时保证了服务恢复后的及时重连。

**3) Shell 子进程管理**

Client 启动一个交互式 shell 子进程，并将其 stdin/stdout 与 WebSocket 双向桥接。支持两种模式：

**PTY 模式（默认）**：使用伪终端，支持 vim/top/htop 等 TUI 程序，输出以二进制帧传输：

```python
master_fd, slave_fd = pty.openpty()
_set_winsize(master_fd, 50, 200)
shell_proc = subprocess.Popen(
    [shell, "-i"],
    stdin=slave_fd, stdout=slave_fd, stderr=slave_fd,
    preexec_fn=os.setsid,
)
```

PTY 模式的关键优势：
- 支持 TUI 程序（vim、top、htop）
- 支持 `__RESIZE:rows,cols` 动态调整窗口大小
- 支持 `__SIGNAL:SIGINT` 等信号转发
- 二进制帧传输，完整保留终端输出格式
- 禁用 ECHO 回显，避免前端看到双重输入

**管道模式（`--no-pty`）**：向后兼容，使用普通管道和行缓冲输出：

```python
shell_proc = subprocess.Popen(
    [shell, "-i"],           # 交互式模式
    stdin=subprocess.PIPE,   # 命令输入
    stdout=subprocess.PIPE,  # 标准输出
    stderr=subprocess.STDOUT,# stderr 重定向到 stdout
    bufsize=0,               # 无缓冲
)
```

后端注册时通过 mode 标记告知 Relay 当前使用的模式：`IAM_BACKEND:<token>:<name>:pty` 或 `IAM_BACKEND:<token>:<name>:pipe`。Relay 在 `LIST` 输出中显示每个后端的模式，前端可据此选择合适的交互方式。

**4) 行缓冲输出转发**

Shell 输出的转发采用了智能行缓冲策略：

```python
def read_and_forward():
    buf = bytearray()
    while True:
        byte = shell_proc.stdout.read(1)  # 逐字节读取
        if not byte:
            break
        buf.extend(byte)
        # 遇到换行符或缓冲区达到 4096 字节时发送
        if byte == b"\n" or len(buf) >= 4096:
            ws.send(buf.decode("utf-8", errors="replace"))
            buf.clear()
```

这个设计平衡了实时性和效率：
- **逐字节读取**确保不会遗漏任何输出
- **行边界优先**保证输出按自然行分割，前端显示整洁
- **4096 字节上限**防止长行（如 base64 编码的文件内容）导致发送延迟
- **UTF-8 + errors="replace"** 容错处理二进制输出

**5) 心跳保活线程**

独立的守护线程每 30 秒发送一次心跳：

```python
def _heartbeat(ws, reconnect_event):
    while not reconnect_event.is_set():
        try:
            ws.send("__PING__")
            time.sleep(_HEARTBEAT_INTERVAL)
        except Exception:
            reconnect_event.set()  # 触发重连
            break
```

使用 `threading.Event` 作为线程间通信机制——心跳线程检测到发送失败时设置事件，主线程在下一次循环时感知到重连信号。这比轮询 `ws.connected` 更高效，比强制关闭连接更优雅。

**6) 线程模型**

Client 的线程模型取决于运行模式：

**PTY 模式**使用三线程模型：

```
┌──────────────────────────────────────────┐
│              Client 进程                  │
│                                          │
│  ┌─────────┐  ┌──────────┐  ┌────────┐  │
│  │ 主线程   │  │ 读取线程  │  │ 心跳   │  │
│  │         │  │          │  │ 线程   │  │
│  │ ws.recv │  │ stdout   │  │ PING   │  │
│  │   ↓     │  │   read   │  │ 每30s  │  │
│  │ stdin   │  │   ↓      │  │        │  │
│  │ write   │  │ ws.send  │  │        │  │
│  └─────────┘  └──────────┘  └────────┘  │
│       │            │             │       │
│       ▼            ▼             ▼       │
│  ┌──────────────────────────────────┐    │
│  │     WebSocket 连接               │    │
│  │     (websocket-client)           │    │
│  └──────────────────────────────────┘    │
│                  │                       │
│                  ▼                       │
│  ┌──────────────────────────────────┐    │
│  │     Shell 子进程 (bash -i)       │    │
│  └──────────────────────────────────┘    │
└──────────────────────────────────────────┘
```

- **主线程**：从 WebSocket 接收命令，写入 shell 的 stdin
- **读取线程**：从 shell 的 stdout 读取输出，发送到 WebSocket
- **心跳线程**：定期发送 `__PING__` 保持连接活跃

三线程模型解决了 shell I/O 的双工问题——bash 的 stdin 和 stdout 是两个独立的文件描述符，需要同时读写，单线程模型无法处理。虽然可以使用 `select`/`poll` 实现 I/O 多路复用，但线程模型代码更直观、更容易理解和维护。

**管道模式**使用相同的三线程结构，区别在于：读取线程从 `stdout.read(1)` 逐字节读取并按行缓冲，而非 PTY 模式下使用 `select` + `os.read` 批量读取二进制数据。

#### 2.3.3 CLI（命令行接口）—— `cli.py`

CLI 层使用 Click 框架提供两个子命令：`relay` 和 `client`。

**设计亮点**：

**1) 环境变量支持**

Token 可以通过 `WS_TUNNEL_TOKEN` 环境变量设置，避免在命令行中暴露敏感信息：

```python
_DEFAULT_TOKEN = os.environ.get("WS_TUNNEL_TOKEN", None)

@click.option("--token", "-t", default=_DEFAULT_TOKEN,
    help='认证令牌。也可通过 WS_TUNNEL_TOKEN 环境变量设置。')
```

这个设计在 systemd 服务部署场景中特别有用——token 存储在 service 文件的 `Environment=` 指令中，不出现在进程参数列表里，也不会被 `ps aux` 泄露。

**2) 分级日志控制**

```python
def _setup_logging(verbose: bool, quiet: bool):
    if verbose:
        level = logging.DEBUG
    elif quiet:
        level = logging.WARNING
    else:
        level = logging.INFO
```

三级日志满足不同场景：`--verbose` 用于调试，默认 INFO 用于正常运维，`--quiet` 用于后台服务只关注异常。

**3) 双入口设计**

同时支持两种运行方式：

```bash
# 方式一：CLI 命令
ws-tunnel relay --port 8080

# 方式二：模块运行
python -m ws_tunnel relay --port 8080
```

`__main__.py` 的存在让包可以作为模块直接运行，这在未安装到 PATH 的环境（如直接 clone 源码）中非常方便。

### 2.4 认证协议设计

ws-tunnel 使用了一个极简但有效的认证协议：

**握手流程**：

```
Backend:                    Relay:                    Frontend:
  │                           │                           │
  │── IAM_BACKEND:token:name ─►│                           │
  │                           │── (注册成功) ──────────────│
  │                           │                           │
  │                           │◄── AUTH:token ────────────│
  │                           │── AUTH_OK ───────────────►│
  │                           │                           │
  │◄── 命令 ──────────────────│◄── ls -la ────────────────│
  │── 输出结果 ──────────────►│── 输出结果 ──────────────►│
```

**协议细节**：

1. 所有连接建立后，必须在 30 秒内发送认证消息
2. 后端消息格式：`IAM_BACKEND:<token>:<name>:<pty|pipe>`（token、name、mode 均可选）
3. 前端消息格式：`AUTH:<token>`
4. 前端也支持 URL 参数认证：`ws://host:port?token=<token>`，连接后自动返回 `AUTH_OK`，无需手动发送 AUTH 消息
5. 认证失败会收到 `AUTH_FAIL` 并被断开（状态码 1008）
6. 不设 token 时保持向后兼容——前端发任何内容都通过认证

**安全考量**：

- Token 以明文形式在 WebSocket 消息中传输。在 wss:// 加密通道下这是安全的；在 ws:// 下则存在中间人窃听风险
- 30 秒认证超时防止资源耗尽攻击
- 拒绝重复后端名称注册（状态码 1008）
- 认证失败的连接立即关闭，不占用资源

### 2.5 数据流完整路径

让我们跟踪一个完整的命令执行周期，理解数据如何在三个组件间流转：

```
时间线  Frontend              Relay                 Backend
  │                           │                        │
  t1 ── ws.connect ──────────►│                        │
  │                           │                        │
  t2 ── "AUTH:secret" ───────►│                        │
  │                           │── 验证 token           │
  │◄── "AUTH_OK" ─────────────│                        │
  │                           │                        │
  t3 ── "ls -la" ────────────►│                        │
  │                           │── 查找目标后端         │
  │                           │── "ls -la" ───────────►│
  │                           │                        │── bash: ls -la
  │                           │                        │── 输出: file list
  │                           │◄── "total 32\n..." ────│
  │◄── "total 32\n..." ──────│                        │
  │                           │                        │
  t4 ── ws.close ────────────►│                        │
```

关键路径分析：
- **Frontend → Backend**：命令经过一次中继转发，延迟 = 1 个网络 RTT
- **Backend → Frontend**：输出经过一次中继广播，延迟 = 1 个网络 RTT
- **总延迟**：命令执行一去一回约 2 个 RTT（Frontend↔Relay + Relay↔Backend）

## 三、技术特色与工程亮点

### 3.1 异步/同步混合架构

ws-tunnel 最引人注目的技术决策之一是：Relay 使用异步（`websockets`），Client 使用同步（`websocket-client`）。

这不是随意的选择，而是基于对两个组件运行环境的深刻理解：

**Relay（VPS 端）为什么用异步？**
- VPS 需要同时处理多个前端和后端连接
- 异步 I/O 天然适合这种"多连接、低 CPU、高 I/O 等待"的场景
- `websockets` 库基于 asyncio，性能优异，API 简洁
- 一个线程处理所有连接，无需线程同步开销

**Client（容器端）为什么用同步？**
- `websocket-client` 库原生支持 `http_proxy_host` 参数
- 容器端只需维护一个 WebSocket 连接，异步没有优势
- 同步模型 + 线程的代码更直观，调试更容易
- 最关键：`websockets`（异步库）不原生支持 HTTP 代理隧道！

这个决策体现了工程师的务实精神——不追求技术栈的一致性，而是为每个组件选择最适合的工具。

### 3.2 闭包 vs 类：状态管理的设计哲学

Relay 的核心 handler 使用闭包管理共享状态，而非传统的面向对象设计：

```python
def _make_handler(token):
    backends: dict = {}  # 共享状态
    frontends: set = set()

    async def handler(websocket):
        nonlocal backends, frontends
        # ...

    return handler
```

这种设计有其深刻的美学考量：

**闭包的优势**：
- 所有状态变量集中定义在一处，关系一目了然
- 状态的生命周期与 handler 绑定，不存在"忘了初始化"的风险
- 代码量更少，没有 `self.` 前缀的噪音
- Python 的 `nonlocal` 关键字让闭包可变状态管理清晰明了

**何时应该用类？**
如果 Relay 需要更复杂的状态管理（如定时任务、外部 API 调用、配置热重载），类会更合适。但对于当前规模——一个 handler 函数加两个集合——闭包是完美的选择。

### 3.3 防御性编程实践

代码中处处体现着防御性编程的思想：

**1) 死连接清理**

每次向前端/后端发送消息时，都会捕获异常并清理断开的连接：

```python
dead = set()
for f in frontends:
    try:
        await f.send(payload)
    except Exception:
        dead.add(f)
frontends -= dead
```

不能在遍历集合时修改它，所以使用临时集合 `dead` 收集失效连接，遍历结束后统一移除。

**2) finally 清理保障**

```python
try:
    # ... 处理连接 ...
except asyncio.TimeoutError:
    await websocket.close(1002, "Protocol error")
except websockets.exceptions.ConnectionClosed:
    pass
except Exception as e:
    logger.exception(f"Handler error: {e}")
finally:
    # 无论如何都清理
    for n, ws in list(backends.items()):
        if ws == websocket:
            backends.pop(n, None)
    frontends.discard(websocket)
```

`finally` 块确保无论连接因何种原因退出（正常断开、超时、异常、服务端主动关闭），共享状态都会被正确清理，不会出现"僵尸连接"泄漏。

**3) 心跳线程的安全退出**

```python
reconnect_event = threading.Event()

# 心跳线程
def _heartbeat(ws, reconnect_event):
    while not reconnect_event.is_set():
        try:
            ws.send("__PING__")
            time.sleep(_HEARTBEAT_INTERVAL)
        except Exception:
            reconnect_event.set()
            break

# 主线程断开时
finally:
    reconnect_event.set()  # 通知心跳线程退出
    shell_proc.kill()      # 终止 shell 子进程
    ws.close()             # 关闭 WebSocket 连接
```

`threading.Event` 是线程安全的信号机制，心跳线程和主线程通过它协调退出。主线程在任何退出路径（正常、异常、超时）都会设置事件，心跳线程在下一次检查时感知到并退出，避免了孤儿线程。

### 3.4 向后兼容的艺术

ws-tunnel 在功能迭代过程中始终保持着严格的向后兼容：

**1) Token 可选性**

不设 token 时，认证逻辑完全旁路：

```python
def _is_frontend_auth(msg, token):
    if token:
        return msg == f"AUTH:{token}"
    return True  # 无 token 时，任何消息都通过
```

这意味着旧版客户端（不发送 `AUTH:` 前缀）在新版 Relay 上仍能正常工作。

**2) 命名可选性**

后端名称也是可选的，自动命名机制保证无名称时也能工作：

```python
_backend_counter += 1
return f"backend-{_backend_counter}"
```

**3) 标签可选性**

多后端输出标签只在后端数量大于 1 时添加：

```python
await _forward_to_frontends(
    frontends, message,
    name if len(backends) > 1 else None  # 单后端时不加标签
)
```

**4) 默认路由**

前端不使用 `@name` 语法时，命令自动路由到第一个后端：

```python
if backends:
    name, ws_backend = next(iter(backends.items()))
    await ws_backend.send(msg)
```

这种层层兼容的设计使得 ws-tunnel 的每次升级都是无感的——旧的部署不需要任何修改就能享受到新版本的稳定性改进，同时新功能对有需要的用户是可选的。

### 3.5 代码组织与模块划分

核心代码分布在 4 个文件中（不含 `__main__.py`）：

```
relay.py    442 行 (55%)  — 核心中继逻辑（多后端路由、认证、PTY 转发）
client.py   339 行 (42%)  — 客户端连接与 Shell 管理（PTY/管道双模式）
cli.py       95 行 (12%)  — 命令行接口
__init__.py  10 行 ( 1%)  — 包导出
__main__.py   9 行 ( 1%)  — 模块运行支持
```

每个文件都有明确单一的职责：

- `relay.py`：只管 WebSocket 连接的路由和转发
- `client.py`：只管与 Relay 的连接和 shell 的桥接
- `cli.py`：只管参数解析和日志配置
- `__init__.py`：只管包级别的 API 导出

没有交叉依赖，没有循环引用，每个模块都可以独立测试和理解。

## 五、网络穿透原理深度解析

### 5.1 HTTP CONNECT 隧道原理

ws-tunnel 穿透能力的核心在于 HTTP CONNECT 方法。理解这一点是理解整个项目的关键。

**标准 HTTP 代理流程**：

```
Client → Proxy: GET http://example.com/page HTTP/1.1
Proxy → Server: GET /page HTTP/1.1 (Host: example.com)
Server → Proxy: 200 OK (content)
Proxy → Client: 200 OK (content)
```

在标准模式下，代理服务器理解 HTTP 协议，会解析请求并重新发起。这意味着代理可以审查、过滤请求内容。

**CONNECT 隧道模式**：

```
Client → Proxy: CONNECT target-vps:8080 HTTP/1.1
Proxy → Client: HTTP/1.1 200 Connection Established
# 此后 Proxy 透明转发所有字节
Client ↔ Proxy ↔ Target: (任意 TCP 数据流)
```

CONNECT 方法请求代理建立一条到目标服务器的 TCP 隧道。一旦隧道建立，代理不再检查隧道中的数据内容，只是透明转发字节流。这就是为什么 WebSocket 握手（本质是一个 HTTP Upgrade 请求）能通过 CONNECT 隧道成功——代理看到的是一个 HTTP CONNECT 请求，隧道内的 WebSocket 帧对代理是透明的。

### 5.2 WebSocket 握手伪装

WebSocket 连接建立过程本身就伪装成了 HTTP 请求：

```
GET / HTTP/1.1
Host: target-vps:8080
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

从网络审查的角度看，这看起来就是一个普通的 HTTP GET 请求（带有几个不常见的头）。只有在仔细检查 `Upgrade: websocket` 头时才能识别为 WebSocket，而很多企业级防火墙和 DPI 系统不会对 HTTP CONNECT 隧道内的流量做这种深度检查。

### 5.3 网络路径分析

完整的网络路径分为两段：

**容器 → VPS（受限路径）**：

```
容器进程
  ↓ websocket-client.send()
WebSocket 帧（二进制）
  ↓ HTTP CONNECT 隧道
HTTP 代理 (127.0.0.1:18080)
  ↓ TCP 连接
VPS (公网 IP:8080)
  ↓ websockets 库接收
Relay 进程
```

**操作者 → VPS（自由路径）**：

```
Frontend 进程
  ↓ WebSocket 连接
VPS (公网 IP:8080)
  ↓ websockets 库接收
Relay 进程
```

第一段路径是项目存在的意义——突破"只能 HTTP 出站"的限制。第二段路径没有任何限制，操作者可以使用任何 WebSocket 客户端。

### 5.4 为什么其他方案不行？

让我们分析主流内网穿透工具为什么在"仅 HTTP 代理出站"的场景下失效：

| 工具 | 失效原因 |
|------|---------|
| **SSH** | 需要 TCP 直连，无法走 HTTP 代理 |
| **ngrok** | 客户端不支持 HTTP 代理，且依赖第三方服务器 |
| **cloudflared** | 使用 QUIC 协议（基于 UDP），且需 Cloudflare 账户 |
| **frp** | 客户端不支持 HTTP CONNECT 代理 |
| **chisel** | 虽可配置代理，但需要编译 Go 二进制文件，容器环境可能没有权限 |

ws-tunnel 的独特优势在于：**纯 Python 实现 + 原生 HTTP 代理支持 + 单文件可部署**。

## 六、安全设计分析

### 6.1 安全层次模型

ws-tunnel 的安全设计可以分四个层次理解：

```
┌───────────────────────────────────┐
│  L4: 操作安全                      │  Token 不入命令行历史
├───────────────────────────────────┤
│  L3: 传输安全                      │  TLS/WSS 加密通道
├───────────────────────────────────┤
│  L2: 协议安全                      │  30 秒认证超时、连接关闭
├───────────────────────────────────┤
│  L1: 认证安全                      │  Token 认证协议
└───────────────────────────────────┘
```

### 6.2 认证机制

Token 认证是最基本的安全屏障：

- Token 通过 `--token` 参数或 `WS_TUNNEL_TOKEN` 环境变量配置
- 环境变量方式避免了 token 出现在进程参数列表中（`ps aux` 不可见）
- 后端和前端使用不同的认证消息格式，防止角色混淆
- 认证失败立即关闭连接（状态码 1008），不消耗服务端资源

### 6.3 传输加密

三种 TLS 部署模式满足不同安全需求：

| 模式 | 安全级别 | 适用场景 |
|------|---------|---------|
| ws://（无加密） | ❌ 明文 | 仅内网测试 |
| wss:// 自签名 | ⚠️ 加密但需信任自签名证书 | 开发测试 |
| wss:// + CA 证书 | ✅ 完整 TLS | 生产环境 |
| nginx 反向代理 + TLS | ✅ 最佳实践 | 生产环境（推荐） |

nginx 反向代理模式的优势在于：证书续期（certbot）不影响 ws-tunnel 进程，ws-tunnel 不需要证书文件权限，重启 nginx 即可更新证书。

### 6.4 潜在安全改进方向

当前设计在以下方面仍有改进空间：

1. **Token 明文传输**：在 ws:// 模式下，token 可被中间人截获。建议增加 challenge-response 认证
2. **无连接速率限制**：恶意客户端可以高频发送连接请求。建议增加 IP 级别的连接频率限制
3. **无审计日志**：当前只有运行日志，缺少结构化的命令审计记录
4. **无会话超时**：已认证的前端连接没有空闲超时机制

## 七、性能分析

### 7.1 资源消耗

**内存占用估算**：

| 组件 | 基础内存 | 峰值内存 |
|------|---------|---------|
| Relay 进程 | ~15MB | ~25MB（多连接时） |
| Client 进程 | ~10MB | ~20MB（含 bash 子进程） |
| **总计** | **~25MB** | **~45MB** |

对比：
- ngrok 客户端：~20MB
- cloudflared：~30MB
- frp 客户端：~10MB

ws-tunnel 的内存占用处于合理范围，且没有额外的 Go runtime 或 Electron 开销。

### 7.2 延迟分析

命令执行的一个完整往返延迟：

```
Frontend 发送 → [网络 RTT₁] → Relay 接收
Relay 转发 → [本地处理 <1ms] → Backend 接收
Backend 执行命令 → [Shell 执行时间] → 输出就绪
Backend 发送输出 → [网络 RTT₂] → Relay 接收
Relay 广播 → [本地处理 <1ms] → Frontend 接收
```

总延迟 ≈ RTT₁ + RTT₂ + Shell 执行时间

在典型 VPS 部署中（Frontend 延迟 50ms，Backend 延迟 100ms），一个简单命令（如 `echo hello`）的往返延迟约 150-200ms，对交互式 Shell 来说完全可接受。

### 7.3 吞吐量分析

WebSocket 的理论吞吐量远超 Shell 交互需求：

- WebSocket 帧开销：2-14 字节/帧
- 典型 Shell 命令输出：100-10000 字节
- 即使是 `cat large_file` 场景，行缓冲（4096 字节/消息）的效率也足够

行缓冲设计使得 ws-tunnel 在大输出场景下的网络效率接近最优——既不会逐字节发送浪费帧头开销，也不会等到缓冲区满才发送影响实时性。

### 7.4 并发连接容量

由于 Relay 基于 asyncio 异步架构，单进程即可处理大量并发连接：

- 前端连接：每个连接占用极少量内存（WebSocket 对象 + 接收缓冲区）
- 后端连接：每个连接额外包含一个 bash 子进程的输出缓冲
- 理论容量：数百个前端 + 数十个后端（受限于 VPS 内存）

在实际使用中，远程 Shell 场景通常只有 1-3 个后端和 1-5 个前端，远未触及性能天花板。

### 8.1 已实现的扩展能力

### 8.2 潜在扩展方向

**1) 通用 TCP 隧道**

将 shell 子进程替换为 TCP socket 连接，即可转发任意 TCP 流量：

```python
# 当前：Shell 隧道
shell_proc = subprocess.Popen([shell, "-i"], ...)

# 扩展：TCP 隧道
tcp_conn = socket.create_connection(("localhost", 3306))  # MySQL
```

这可以将 ws-tunnel 演变为一个支持 HTTP 代理穿透的通用 TCP 隧道工具，功能上类似 chisel，但代码量少一个数量级。

**2) SOCKS5 代理**

在 Frontend 端实现 SOCKS5 协议，将 ws-tunnel 变成一个完整的代理链：

```
应用 → SOCKS5 → Frontend → WebSocket → Relay → Backend → 目标
```

**3) 文件传输**

通过特殊命令前缀实现文件上传/下载：

```
@@UPLOAD:/path/to/file:base64_data@@
@@DOWNLOAD:/path/to/file@@
```

**4) 结构化消息协议**

将纯文本消息升级为 JSON 结构化消息，支持元数据、类型标记、确认机制等：

```json
{"type": "command", "id": "uuid", "body": "ls -la", "target": "backend-1"}
{"type": "output", "id": "uuid", "body": "total 32\n...", "stream": "stdout"}
{"type": "ack", "id": "uuid"}
```

**5) 会话录制**

在 Relay 端记录所有命令和输出，用于审计和回放：

```python
async def _forward_to_frontends(frontends, message, tag=None):
    # 审计日志
    audit_log.write(f"{time.time()},{tag},{message}")
    # 正常转发
    ...
```

### 8.3 架构的可扩展边界

当前架构在以下方向上存在扩展瓶颈：

1. **单进程限制**：Relay 是单进程 asyncio 应用，无法利用多核 CPU。在高吞吐场景下可能需要多进程部署
2. **无持久化**：所有状态（后端列表、前端集合）都在内存中，进程重启后丢失。需要 Redis/数据库来持久化
3. **无负载均衡**：多个前端连接同一个后端时，所有命令串行执行。需要队列机制
4. **无服务发现**：后端名称硬编码或自动生成，没有与服务发现系统（Consul、etcd）的集成

这些瓶颈在当前的远程 Shell 场景中不是问题，但在向"通用隧道平台"演进时需要解决。

### 10.2 核心差异化优势

**1) HTTP 代理穿透能力**

这是 ws-tunnel 最大的护城河。在"仅允许 HTTP 代理出站"的网络环境中，几乎所有竞品都失效，而 ws-tunnel 原生支持。

**2) 零编译部署**

纯 Python 实现，无需编译。在目标容器中：

```bash
pip install websocket-client  # 或直接下载 .whl 文件
python3 client.py --server ws://... --proxy http://...
```

对比需要上传 Go 二进制文件的方案（frp、chisel），在文件系统受限或架构不匹配的环境中优势明显。

**3) 多前端广播**

一个后端的输出可以同时广播给多个前端。这在以下场景中极为有用：

- 多人同时观察同一容器的操作
- 操作过程中的实时审计
- 教学/演示场景

**4) 协议透明性**

所有代码都是 Python，任何开发者都能在几分钟内理解完整的协议和逻辑。对比 Go 编写的 frp 和 chisel（需要理解 Go 的并发模型和网络库），理解成本极低。

### 10.3 适用场景评估

| 场景 | 推荐工具 | 原因 |

| IoT 设备管理 | **ws-tunnel** | 极低资源占用 |
| 安全测试/渗透 | **ws-tunnel** | 流量伪装为 HTTP |

ws-tunnel 在自己的核心场景（受限网络穿透 + 轻量级远程 Shell）中具有不可替代的优势。

## 十一、代码美学与编程哲学

### 11.1 极简主义的践行

ws-tunnel 的 536 行代码做到了：

- **一个完整的远程 Shell 系统**：连接管理、认证、路由、转发、心跳、重连
- **生产级可靠性**：异常处理、资源清理、状态恢复
- **用户体验**：CLI 参数设计、日志分级、故障排查指南
- **可扩展性**：多后端、可插拔 Shell、库 API

这种密度来自于对问题本质的深刻理解——知道哪些功能是核心必须的，哪些可以延后，哪些完全不需要。

### 11.2 "刚好够用"的工程哲学

代码中没有过度设计：

- 没有引入消息队列，因为 Shell 交互的命令速率很低
- 没有实现心跳的指数退避，因为固定间隔足够
- 没有使用 ORM 或数据库，因为状态就是两个集合
- 没有引入配置文件，因为参数够少

每一行代码都有存在的理由，没有一行是为了"将来可能用到"而写的。

### 11.3 注释的艺术

代码注释遵循了"为什么而非是什么"的原则：

```python
# 好注释：解释设计意图
# ── 心跳 ──  (标记功能区域)
# 遇到换行符或缓冲足够大时发送  (解释策略)
# 不设 token 时保持向后兼容  (解释设计决策)

# 而非：
# i += 1  # 自增
```

这种注释风格让代码读起来像一篇叙事文——每个段落有明确的主题，每个设计决策都有解释。

### 11.4 函数式思维的渗透

虽然使用 Python 这种多范式语言，代码中处处体现着函数式思维：

- **闭包状态管理**（`_make_handler`）
- **纯函数**（`_parse_backend_auth`、`_is_frontend_auth`）
- **不可变性**（使用集合的差集操作而非原地修改）
- **高阶函数模式**（handler 作为参数传递给 `websockets.serve`）

这些模式让代码更紧凑、更易推理、更少副作用。

### 命令速查

```bash
# 安装
pip install wsstunnel

# 启动中继（VPS 端）
ws-tunnel relay --port 8080 --token mysecret

# 启动客户端（容器端，PTY 模式）
ws-tunnel client --server ws://vps:8080 --proxy http://127.0.0.1:18080 --token mysecret --name my-container

# 启动客户端（容器端，管道模式）
ws-tunnel client --server ws://vps:8080 --proxy http://127.0.0.1:18080 --token mysecret --no-pty

# 前端连接（手动认证）
websocat ws://vps:8080
# 输入 AUTH:mysecret 认证
# 输入 LIST 查看后端列表
# 输入 USE <name> 切换默认后端
# 输入 @name command 临时发送给指定后端

# 前端连接（URL 自动认证）
websocat "ws://vps:8080?token=mysecret"
# 自动完成认证，直接开始操作
```

### 文件结构

```
ws_tunnel/
├── __init__.py     # 导出 run_relay, run_client
├── __main__.py     # python -m ws_tunnel 支持
├── cli.py          # Click CLI：relay + client 子命令
├── client.py       # 后端客户端：WebSocket + Shell 桥接

```

### 关键参数

| 参数 | 组件 | 说明 |
|------|------|------|

### 消息类型

| 方向 | 消息 | 含义 |
|------|------|------|
| Backend → Relay | `IAM_BACKEND:<token>:<name>:<mode>` | 后端注册（mode = pty/pipe） |
| Frontend → Relay | `AUTH:<token>` | 前端认证 |
| Frontend → Relay | 连接 URL `?token=<token>` | URL 参数认证（自动完成） |
| Relay → Frontend | `AUTH_OK` / `AUTH_FAIL` | 认证结果 |
| Frontend → Relay | `LIST` | 查询后端列表 |
| Frontend → Relay | `USE <name>` | 切换默认目标后端 |
| Frontend → Relay | `USE` | 查看当前目标后端 |
| Frontend → Relay | `@<name> <cmd>` | 命名路由命令（临时） |
| Frontend → Relay | `<cmd>` | 默认路由命令 |
| Frontend → Relay | `__RESIZE:rows,cols` | PTY 窗口大小调整 |
| Frontend → Relay | `__SIGNAL:SIGXXX` | 向后端进程组发送信号 |
| Frontend → Relay | 二进制帧 | 原始按键输入（PTY 模式） |
| Backend → Relay | `__PING__` | 心跳探测 |
| Relay → Backend | `__PONG__` | 心跳响应 |
| Backend → Relay | `<output>` | Shell 文本输出 |
| Backend → Relay | 二进制帧 | PTY 原始输出 |
